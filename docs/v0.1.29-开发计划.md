# v0.1.29 开发计划

## 1. 版本主题

补齐常见降级场景的明确反馈，让用户在 fake 模式、缺少 API Key 或最小路径中断时，能更快知道自己卡在了哪里。

## 2. 为什么现在做

`v0.1.28` 已经把最小 happy path 收口成固定路径，下一步最自然的问题就是“如果这条路径没跑通，用户能不能马上知道为什么”。这一版应该开始补可用版所需的稳定性反馈，而不是再扩更多示例命令。

## 3. 计划变更

- 为常见 fake 模式或缺配置场景补齐更明确的提示
- 让最小 happy path 中断时的反馈更容易理解
- README 补充对应排障说明
- 增加对应测试或 smoke 验证

## 4. 本版不做

- 自动联网探测
- 安装器级别修复动作
- 复杂交互式故障诊断树

## 5. 验收标准

- [x] 常见降级场景已有更明确的 CLI 反馈
- [x] README 已补充对应排障说明
- [x] 相关测试或 smoke 验证补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.29` 可以判定为已完成，CLI 现在已经能在 fake 模式、缺少 API Key 导致的降级模式，以及“单轮命令看起来没输出”的场景里直接给出明确提示，帮助用户把“当前是降级模式”与“系统没有响应”区分开。

完成依据：

- 单轮命令在 fake 模式下会直接提示当前处于降级模式
- fake 模式下出现空输出时，CLI 会明确说明这是 fake client 的预期行为
- REPL 启动时会更清楚提示当前是否处于 fake 模式
- README 已补充对应排障说明
- 相关测试与 smoke 验证已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前反馈仍然以本地提示为主，不会自动修复配置问题
- 当前不会探测外部网络或 API 实际连通性
- 下一步应进入 `v0.1.30` 的交付收口，补齐最终验证清单与可用版说明

## 7. 风险与停手线

- 如果一口气把反馈逻辑做成复杂诊断系统，会推高 `v0.1.30` 前的实现成本
- 如果只继续写 happy path 不补中断反馈，用户仍然可能在关键节点卡住
- 因此本版应控制在“先把常见卡点说明白”而不是“实现完整故障专家系统”
