# v0.1.28 开发计划

## 1. 版本主题

补齐一条最小 happy path，让用户在完成 `assistant check` 后，能按 README 的固定顺序跑通一次真实可感知的最小使用闭环。

## 2. 为什么现在做

`v0.1.27` 已经先把“当前环境能不能跑”这个问题显式化，下一步最自然的问题就是“确认能跑之后，用户第一条该怎么用”。这一版应该开始收口最小使用路径，而不是继续分散在零散示例上。

## 3. 计划变更

- 为 README 补齐更明确的最小 happy path 顺序
- 收口一条从自检到单轮任务、再到任务/审计入口的最小使用路径
- 让 CLI 输出或文档中的下一步提示更贴近这条固定路径
- 增加对应测试或 smoke 验证

## 4. 本版不做

- 复杂业务场景模板
- 多工作流向导
- 发布级安装器

## 5. 验收标准

- [x] README 已补齐一条固定的最小 happy path
- [x] CLI 或相关文档的下一步提示已与最小路径对齐
- [x] 相关测试或 smoke 验证补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.28` 可以判定为已完成，CLI 自检输出和 README 现在已经对齐到同一条固定的 `minimal_happy_path`，用户在完成 `assistant check` 后，可以按同一组步骤继续理解工具边界、验证任务入口，并拿到审计入口导航。

完成依据：

- README 已补齐固定的最小 happy path
- `assistant check` 输出已新增 `minimal_happy_path`
- 相关测试与 smoke 验证已补充，并通过 `go test ./...`
- CLI 的下一步提示已与最小路径对齐

### 6.2 当前边界

- 当前最小 happy path 仍然是文档和提示层收口，不是完整初始化向导
- 当前 fake 模式仍以 dry run 和本地验证为主，不等价于真实模型能力
- 下一步应继续补“常见失败时的明确反馈”，避免用户在 fake 或缺配置场景里误判系统状态

## 7. 风险与停手线

- 如果一口气把所有示例都塞进 happy path，会让最小路径再次失焦
- 如果只写文档不收口入口提示，用户仍然不知道第一步该按哪条线走
- 因此本版应控制在“明确一条最小路径”而不是“罗列更多可能路径”
