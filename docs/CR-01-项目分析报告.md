## 一、项目概述

**icoo_assistant** 是一个基于 Go 语言开发的 **CLI 编码助手智能体**，通过 Anthropic Claude API 驱动。它在本地工作区目录中运行，接收自然语言查询，使用基于工具的智能体循环来读写文件、执行 Shell 命令、管理后台任务、跟踪待办事项、委托子智能体执行任务，并审计运行时事件。

| 属性 | 值 |
|------|-----|
| 项目名称 | icoo_assistant |
| 当前版本 | 0.1.36 |
| 开发语言 | Go 1.23.0 |
| 运行模式 | CLI（REPL + 单次运行） |
| 核心依赖 | anthropic-sdk-go v1.38.0 |
| 默认模型 | claude-opus-4-7 |

## 三、架构设计

### 3.1 分层架构

```
┌──────────────────────────────────────┐
│         CLI (cmd/assistant)          │
│  main.go → app.go (组件装配)          │
├──────────────────────────────────────┤
│    Agent Runner (internal/agent)     │
│  主运行循环：LLM ↔ 工具 的迭代      │
├──────────────┬───────────────────────┤
│   LLM Client │   Tool Registry       │
│  (Anthropic  │   (12 个工具实现)      │
│   / Fake)    │                       │
├──────────────┴───────────────────────┤
│   支撑服务层                          │
│  config | workspace | commandutil    │
│  task | background | compact         │
│  todo | skill | subagent | hooks     │
└──────────────────────────────────────┘
```

### 3.2 核心运行循环

智能体的核心执行流程（`internal/agent/loop.go:42`）：

1. **启动**：接收初始消息列表，生成 RunID，发送 `agent.run.started` 事件
2. **每轮迭代**（最多 `AGENT_MAX_ROUNDS` 轮）：
   - **后台通知轮询**：检查是否有已完成的后台作业，如有则注入通知消息
   - **微压缩**：清除旧工具结果，避免上下文膨胀
   - **自动压缩**：若 token 估算超过阈值，调用 LLM 摘要压缩
   - **LLM 调用**：发送消息到 Claude API（支持流式/非流式）
   - **响应处理**：
     - `stop_reason == "tool_use"` → 执行工具调用，返回结果
     - `stop_reason == "end_turn"` → 返回最终消息
   - **工具执行**：遍历 `ToolUse` 列表，调用对应 Handler
     - 特殊处理：`task` 工具委托给子智能体执行
   - **待办提醒**：每 3 轮未使用 `todo` 工具则注入提醒
   - **手动压缩**：若使用了 `compact` 工具，执行对话摘要
3. **结束**：记录会话转录到 JSON 文件

### 3.3 双模式运行

| 模式 | 条件 | 行为 |
|------|------|------|
| **Real Mode** | 设置了 `ANTHROPIC_API_KEY` | 调用真实的 Anthropic API |
| **Fake Mode** | 未设置 `ANTHROPIC_API_KEY` | 返回空响应，用于本地干运行和配置验证 |

### 4.2 工具注册表设计

```
Registry
  └── defs: map[string]Definition
        ├── Tool (llm.Tool)     → 工具的输入 Schema 和描述
        └── Handler (Handler)   → 工具的实际执行逻辑
```

- 工具按名称字母序排序后传递给 LLM
- 执行时通过名称查找对应的 Handler
- 子智能体注册表不包含 `task` 工具（防止无限递归委派）

## 五、关键模块分析

### 5.1 项目任务管理 (`internal/task`)

**核心数据结构**: `Task` — 包含 ID、标题、状态、阻塞依赖、后台上下文、时间戳等字段。

**状态流转**:
```
pending ──→ in_progress ──→ completed
   │                            │
   └── blocked ─────────────────┘
        （有 BlockedBy 时自动设为 blocked）
```

**关键特性**:
- 任务以 JSON 文件存储在 `.tasks/` 目录
- 支持 `BlockedBy` 依赖链：任务完成时自动解锁下游依赖
- 后台作业生命周期关联：`BackgroundContext` 记录关联的后台作业状态
- 并发安全：所有方法通过 `sync.Mutex` 保护

### 5.2 后台任务管理 (`internal/background`)

**核心数据结构**: `Job` — 包含 ID、命令、状态、输出、错误、时间戳等字段。

**状态流转**:
```
running ──→ completed
      └──→ failed (超时或错误)
```

**关键特性**:
- 异步执行：通过 goroutine 启动命令
- 超时控制：默认 120 秒，可配置
- 通知机制：`PollNotifications()` 返回已完成但未通知的作业
- 生命周期钩子：`BeforeStart` / `AfterFinish` 与任务管理系统联动
- 存储：JSON 文件存储在 `.background/` 目录

### 5.3 LLM 客户端 (`internal/llm`)

**接口定义** (`client.go`):
```go
type Client interface {
    CreateMessage(system string, messages []Message, tools []Tool) (Response, error)
    CreateMessageStream(system string, messages []Message, tools []Tool, handler func(string)) (Response, error)
}
```

**两种实现**:
- `AnthropicClient`：基于 `anthropic-sdk-go`，支持流式输出、思考模式、提示缓存、自定义 Base URL
- `FakeClient`：无 API Key 时使用，返回空响应

### 5.4 对话压缩 (`internal/compact`)

- **微压缩**：每轮自动清除旧的工具结果（保留最近 3 轮）
- **自动压缩**：Token 估算超阈值（默认 50000）时触发 LLM 摘要
- **手动压缩**：通过 `compact` 工具主动触发

### 5.5 事件钩子系统 (`internal/agent/hooks`)

| 事件类型 | 触发时机 |
|----------|----------|
| `agent.run.started` | 运行开始 |
| `agent.run.completed` | 运行成功结束 |
| `agent.run.failed` | 超过最大轮数 |
| `agent.round.started` | 每轮开始 |
| `agent.model.requested` | LLM 调用前 |
| `agent.model.responded` | LLM 响应后 |
| `agent.tool.started` | 工具执行前 |
| `agent.tool.completed` | 工具执行后 |
| `agent.todo.reminder_injected` | 待办提醒注入 |
| `agent.compact.auto_requested` | 自动压缩触发 |
| `agent.compact.auto_completed` | 自动压缩完成 |
| `agent.compact.manual_requested` | 手动压缩触发 |
| `agent.background.notifications_injected` | 后台通知注入 |
| `agent.subagent.started` | 子智能体启动 |
| `agent.subagent.completed` | 子智能体完成 |

事件以 JSONL 格式写入 `.agent-hooks/events.jsonl`。

1. 读取工作目录下的 `.env` 文件
2. 忽略空行和 `#` 注释行

4. 支持双引号和单引号包装值
5. 已有环境变量优先（不被 `.env` 覆盖）
6. 布尔值支持 `1/true/yes/on` 和 `0/false/no/off`

## 八、代码质量评估

### 8.1 优点

1. **清晰的模块划分**：13 个内部包职责单一，边界分明
2. **良好的接口设计**：`Client`、`SubagentRunner`、`LifecycleHooks` 等接口支持依赖注入
3. **并发安全**：`task.Manager` 和 `background.Manager` 均使用 `sync.Mutex` 保护
4. **事件驱动审计**：完整的钩子系统记录所有关键运行时事件
5. **双模式设计**：Fake 模式支持无 API Key 的本地干运行
6. **优雅降级**：API Key 未设置时给出清晰的提示和指引
7. **完善的帮助系统**：内置 `--help`、`--version`、`--check`/`doctor` 命令
8. **ID 验证**：任务/作业 ID 使用字符白名单（`[a-zA-Z0-9_-]`）防止路径遍历

### 8.2 可改进点

1. **缺少 README**：项目根目录缺少 README.md 文档
2. **测试覆盖**：仅 `cmd/assistant/` 下有测试文件，`internal/` 包缺少单元测试
3. **所有依赖标记为 indirect**：应明确直接依赖，仅保留传递依赖为 indirect
4. **配置无验证**：加载后不验证配置值的合法性（如 `MaxRounds` <= 0 的边界情况）
5. **错误处理**：部分场景使用 `_` 忽略错误返回值
6. **硬编码值**：如 `KeepRecent: 3`、`MaxBackgroundHistory: 5` 等硬编码在代码中

## 十一、总结

`icoo_assistant` 是一个架构设计良好、功能完整的 CLI 编码助手智能体。其核心亮点在于：

- **模块化分层架构**：CLI → Agent → LLM/Tools → 支撑服务，职责清晰
- **完整的事件审计系统**：覆盖 16 种运行时事件，支持事后分析
- **丰富的工具生态**：12 个工具涵盖文件操作、Shell 执行、任务管理、审计分析等
- **子智能体委派机制**：支持复杂任务的分解与委托执行
- **双模式运行**：Fake/Real 模式切换简单，便于开发调试

当前代码库规模适中（约 13 个内部包），依赖少（仅 1 个核心依赖），维护性良好。建议在未来版本中补充单元测试和 README 文档。
