# v0.0.2 开发计划

## 1. 版本定位

`v0.0.2` 聚焦把当前的单 Agent MVP 从“会话内执行”推进到“项目级任务管理”。核心目标是补齐任务持久化骨架，让系统开始具备跨轮次、跨进程恢复任务状态的能力。

## 2. 版本目标

本版本计划交付以下能力：

- 项目级任务数据结构
- 基于磁盘的任务持久化目录
- 任务创建、读取、更新、列表查询
- `blockedBy` 依赖字段与基础解锁逻辑
- 与现有 `task` / Agent 体系兼容的最小任务管理入口
- 任务系统的基础测试和文档

## 3. 范围定义

### 3.1 纳入 `v0.0.2`

- 新增 `internal/task` 模块
- 新增 `.tasks/` 数据目录约定
- 任务 JSON 结构定义
- Task CRUD
- 基础依赖处理
- 文档与测试补充

### 3.2 暂不纳入 `v0.0.2`

- 后台任务执行器
- 长驻 Team Agent
- 审批协议
- worktree 绑定
- 自主任务认领

## 4. 设计重点

### 4.1 数据模型

任务至少包含以下字段：

- `id`
- `title`
- `status`
- `blockedBy`
- `owner`
- `worktree`
- `createdAt`
- `updatedAt`

状态建议先收敛为：

- `pending`
- `in_progress`
- `completed`
- `blocked`

### 4.2 存储结构

建议沿用仓库内文档已定义的方向：

```text
.tasks/
  task_<id>.json
```

首版优先选择 JSON 文件持久化，不急于引入数据库。

### 4.3 与当前代码的衔接

- 保持现有会话级 `todo` 能力不变
- 新任务系统作为项目级补充，而不是替代 `todo`
- `task` 工具后续可扩展为“委托子代理”与“项目任务管理”两种职责中的其一，`v0.0.2` 先不要强行合并

## 7. 完成度评估

### 7.1 当前结论

`v0.0.2` 可以判定为已完成，任务持久化骨架已经具备后续工具层和团队能力扩展所需的基础结构。

完成依据：

- 已新增 `internal/task` 模块
- 已实现任务创建、读取、列表、更新与依赖解锁
- `NewManager` 会自动初始化 `.tasks/` 目录
- 已补充任务模块测试，`go test ./...` 通过
- README 已补充 `.tasks/` 与任务模块说明

### 7.2 当前边界

- 任务模块当前是内部基础设施，尚未接入正式 CLI 或工具调用入口
- 现有 `task` 工具名仍保留给子代理委托，任务管理入口将在后续版本统一规划
- 当前持久化采用单文件 JSON 方案，暂未处理跨进程竞争写入
## 8. 风险提示

- 任务依赖一旦设计过重，会拖慢后续团队协作模块演进
- 文件持久化需要尽早约束命名和更新策略，避免后续兼容成本
- `task` 命名当前已被子代理委托占用，后续工具层命名需要谨慎演进
