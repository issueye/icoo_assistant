# v0.1.34 开发计划

## 1. 版本主题

在 `.team/` 和 teammate registry 已落地之后，补上 `.team/inbox/` message store，让 Team 主线第一次具备可持久化、可查询的消息承载层。

## 2. 为什么现在做

`v0.1.33` 已经完成了 Team 基础设施的第一步：`.team/config.json` 和 teammate roster 已经可以保存、查询和更新。但只靠 roster 还不足以支撑下一阶段的 lead -> teammate 协作，因为消息还没有落点。

根据路线图，`v0.1.34` 应该进入 Stage B 的第二步，也就是 inbox / message store。当前还不适合直接做真正的 teammate loop 或自动执行，因为在那之前，收发消息本身需要先有一个稳定的持久化承载层。

因此，本版只回答一个问题：如何把 Team 消息写入磁盘并提供最小查询入口。

## 3. 计划变更

- 在 `internal/team` 中增加 `.team/inbox/` 目录与 JSONL 消息存储
- 新增消息结构和 `SendMessage / ListInbox` 能力
- 新增 `team_message` 工具，支持 `send / inbox`
- 将 inbox 目录纳入 `assistant check`
- 更新 README 与工具边界说明，明确当前已经具备 message store，但还没有真实 teammate loop

## 4. 本版不做

- 真正运行中的 lead -> teammate loop
- 收件确认、已读状态和协议 response 跟踪
- 协议审批、自治认领与 worktree 隔离

## 5. 验收标准

- [x] `.team/inbox/` 会自动初始化
- [x] `team_message action=send` 可把消息写入 recipient inbox
- [x] `team_message action=inbox` 可查看某个 recipient 的最近消息
- [x] `assistant check` 能展示 inbox 目录已 ready
- [x] `go test ./...` 通过

## 6. 完成度评估

`v0.1.34` 可以判定为已完成。本版把 Team 主线从“只有 roster”推进到了“有基础消息承载层”，但仍然刻意停在 message store，而不是直接把整个多 Agent 通信链路一口气做满。

本版完成情况如下：

- `internal/team` 新增 `.team/inbox/<recipient>.jsonl` 的消息持久化支持
- `team_message` 工具已支持写入消息和查看收件箱
- 当前消息会校验参与方必须是 `lead` 或已登记 teammate
- `assistant check` 已增加 `team_inbox_dir` 展示
- README 已补充 message store 的定位、示例命令和边界

当前边界判断：

- 这版已经满足 Stage B 第二小步“inbox / message store”
- 还没有实现真正的 lead -> teammate 通信闭环
- 下一版应围绕最小发送与响应链路展开，而不是继续只在 message 字段层做扩展

## 7. 风险与停手线

- 如果继续在 message 结构上补零散字段，会再次退回字段级机械迭代
- 如果现在直接做复杂协议和审批，会绕过最小通信链路这个更基础的问题
- 因此本版停在“消息可写、可查、可持久化”，把下一步空间留给 `v0.1.35` 的最小 lead -> teammate 通信闭环
