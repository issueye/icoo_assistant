# 本地 AI 协议转换网关需求分析文档

## 1. 文档目的

本文档用于定义 `icoo_assistant` 体系内“本地 AI 协议转换网关”的建设需求，明确该网关的产品定位、建设目标、核心场景、功能范围、接口边界与验收口径，为后续架构设计、开发拆解和版本规划提供统一基线。

本文档聚焦的对象不是“Agent 管理控制面”，而是一个部署在本地或单机侧的 AI Gateway。它负责接入平台已提供的：

- Anthropic API
- OpenAI `chat/completions` API
- OpenAI `responses` API

并对外继续提供兼容的：

- Anthropic API
- OpenAI `chat/completions` API
- OpenAI `responses` API

核心目标是让上层客户端只面向一个本地网关地址，即可按原有协议完成调用，而无需直接关心真实上游平台、密钥、协议差异和模型路由策略。

## 2. 建设背景

## 2.1 当前现状

当前项目已经具备：

- 本地 Agent Runtime 骨架
- 独立网关管理服务骨架
- Anthropic 模式下的基础真实模型调用能力
- 本地工具、任务和审计能力

但在模型接入层仍存在明显缺口：

- 上层客户端通常只会实现一种协议，不愿为多个供应商分别适配
- OpenAI `chat/completions` 与 `responses` 正在并存，调用语义不完全一致
- Anthropic 消息结构、停止原因、流式事件和工具调用表达与 OpenAI 系有差异
- 平台可能已经统一采购或代理了上游能力，但调用方仍希望保留原生 SDK / 原生协议使用方式

## 2.2 当前问题

如果没有一个本地协议转换网关，会带来这些问题：

- 每个客户端都要分别适配 Anthropic、OpenAI Chat、OpenAI Responses 三套协议
- 切换模型供应商时，需要修改客户端代码而不是只改网关配置
- 密钥分发会下沉到每个调用端，带来安全和运维成本
- 流式响应、工具调用、错误码和 usage 统计会因供应商不同而难以统一
- 后续想增加审计、限流、模型路由、回退策略时，没有统一接入点

## 2.3 建设目标

建设一个本地 AI 协议转换网关，作为“统一 AI 接入层”，实现：

- 上游多协议接入
- 下游多协议兼容输出
- 同协议透传与跨协议转换
- 统一鉴权、审计、限流、配置和可观测性
- 对现有 `icoo_assistant` Runtime、CLI、未来控制台和外部客户端提供稳定本地入口

## 3. 产品定位

本地 AI 协议转换网关的定位是：

- 本地统一 AI 接入代理
- AI 协议兼容层
- 模型路由与上游适配层
- 密钥隔离与调用治理层
- 审计、限流和回退策略承载层

它不是：

- 新的模型提供方
- 复杂工作流编排引擎
- Agent 控制面替代品
- 完整企业级 API 网关平台

## 4. 目标用户

## 4.1 直接用户

- 本地开发者工具
- `icoo_assistant` Runtime / CLI
- 使用官方 SDK 但希望切到本地统一入口的应用
- 需要兼容不同协议客户端的插件、脚本或桌面程序

## 4.2 间接用户

- 后续 Web 管理端
- 企业内统一模型接入平台
- 自动化任务调度服务

## 5. 核心价值

- 让调用方“按自己熟悉的协议继续调用”
- 把上游差异收敛在网关内部，而不是扩散到业务代码
- 支持协议升级和平滑迁移，例如从 `chat/completions` 迁到 `responses`
- 支持统一审计、监控、限流和成本分析
- 为后续引入模型路由、灰度切换和失败回退提供稳定挂载点

## 6. 范围定义

## 6.1 本期纳入范围

- 对外提供 Anthropic 兼容接口
- 对外提供 OpenAI `chat/completions` 兼容接口
- 对外提供 OpenAI `responses` 兼容接口
- 接入上游 Anthropic / OpenAI Chat / OpenAI Responses 能力
- 支持协议间请求与响应转换
- 支持流式与非流式调用
- 支持模型别名与路由配置
- 支持统一 API Key 校验或本地免密受控模式
- 支持基础 usage 统计、请求日志和错误日志
- 支持健康检查、配置检查和上游连通性检查

## 6.2 暂不纳入范围

- 图片生成、音频转写、向量嵌入等非文本主能力
- 复杂多租户计费系统
- 企业级组织权限中心深度对接
- 分布式高可用集群编排
- 复杂提示词治理平台
- 长链路工作流编排

## 7. 总体业务模型

网关需要同时处理三类对象：

- `northbound protocol`：下游客户端调用网关时使用的协议
- `routing target`：网关决定实际转发到哪个上游协议和模型
- `translation context`：协议转换时保留请求上下文、工具定义、usage、停止原因和流式事件映射的运行上下文

其核心链路为：

1. 客户端以 Anthropic、`chat/completions` 或 `responses` 任一协议请求本地网关。
2. 网关完成鉴权、参数校验、模型别名解析和路由决策。
3. 网关根据路由结果，选择上游 Anthropic、OpenAI Chat 或 OpenAI Responses。
4. 如上下游协议不同，网关执行请求转换。
5. 获取上游响应后，网关执行响应转换并按下游协议返回。
6. 全过程记录审计、usage、错误和性能指标。

## 8. 典型协议转换场景

## 8.1 同协议透传

- 下游使用 Anthropic，网关转发到上游 Anthropic
- 下游使用 `chat/completions`，网关转发到上游 `chat/completions`
- 下游使用 `responses`，网关转发到上游 `responses`

此模式以最小转换成本为目标，主要附加鉴权、路由、审计和限流能力。

## 8.2 跨协议转换

- 下游 Anthropic，实际转发到上游 `responses`
- 下游 `chat/completions`，实际转发到上游 Anthropic
- 下游 `responses`，实际转发到上游 Anthropic

此模式以“兼容调用端协议”与“最大化复用可用上游能力”为目标。

## 8.3 平滑迁移场景

- 旧系统继续发 `chat/completions`
- 新系统逐步切换到 `responses`
- 两者都接入同一本地网关
- 真实上游可先保持不变，后续再逐步切换

## 8.4 统一出口场景

- 本机多个工具都指向 `http://127.0.0.1:<port>`
- 不再让每个工具直接持有真实上游 API Key
- 通过网关集中控制可用模型、上游地址和调用配额

## 9. 协议兼容目标

## 9.1 Anthropic 兼容目标

网关至少应兼容这些能力：

- `POST /v1/messages`
- 流式 messages 输出
- system prompt、messages、多段 content
- `max_tokens`、`temperature`、`top_p`、`stop_sequences`
- tool use / tool result 结构
- usage、stop_reason、id、model 等关键返回字段

## 9.2 OpenAI Chat Completions 兼容目标

网关至少应兼容这些能力：

- `POST /v1/chat/completions`
- 流式 chat completion chunk 输出
- `model`、`messages`、`temperature`、`top_p`、`max_tokens`
- tools / tool_choice
- `response_format` 的基础透传或受限转换
- usage、finish_reason、choices 结构

## 9.3 OpenAI Responses 兼容目标

网关至少应兼容这些能力：

- `POST /v1/responses`
- 流式 responses 事件输出
- `input`、`instructions`、`model`
- tools、tool_choice、并行工具能力的兼容映射
- output item、status、incomplete_details、usage
- 与 Chat 风格消息结构之间的基本互转

## 10. 核心功能需求

## 10.1 P0：基础网关能力

### FR-001 本地服务启动

系统必须支持以本地单进程方式启动，并默认监听本地回环地址。

### FR-002 统一配置加载

系统必须支持从环境变量、配置文件或启动参数加载监听地址、上游地址、密钥、模型映射和日志配置。

### FR-003 健康检查

系统必须提供健康检查接口，至少区分“进程存活”和“上游可用性”两类状态。

### FR-004 协议路由入口

系统必须同时暴露 Anthropic、`chat/completions`、`responses` 三套 HTTP 入口。

### FR-005 请求追踪

系统必须为每次请求生成统一 request id，并贯穿日志、审计和错误响应。

## 10.2 P0：协议识别与转换

### FR-006 下游协议解析

系统必须正确识别不同入口协议的请求结构，并转换成内部统一语义模型。

### FR-007 内部统一语义模型

系统必须在网关内部建立统一的消息、工具、采样参数、usage、停止原因和流式事件模型，避免做点对点硬编码转换。

### FR-008 上游协议适配

系统必须支持将内部统一语义模型映射为上游 Anthropic、OpenAI Chat 或 OpenAI Responses 请求。

### FR-009 响应逆向转换

系统必须支持把不同上游响应转换为下游请求方期待的原生协议格式。

### FR-010 非流式调用

系统必须支持三套协议的标准非流式调用。

### FR-011 流式调用

系统必须支持三套协议的流式调用，并保证事件顺序、结束事件和错误事件可被下游正确消费。

## 10.3 P0：模型路由与映射

### FR-012 模型别名

系统必须支持定义本地模型别名，例如把 `claude-sonnet`、`gpt-4o-chat`、`assistant-default` 映射到真实上游模型。

### FR-013 协议无关路由

系统必须支持“下游协议”和“上游协议”解耦，允许同一个下游模型别名路由到不同上游协议。

### FR-014 默认路由策略

系统必须支持默认路由、按模型路由和按接口类型路由。

### FR-015 回退策略

系统应支持上游失败后的有限回退，例如首选 `responses`，失败时回退到 Anthropic 或 `chat/completions`。

## 10.4 P0：鉴权与安全

### FR-016 下游访问控制

系统必须支持对调用本地网关的客户端进行 API Key 校验，或在本机受控模式下允许免密访问。

### FR-017 上游密钥隔离

系统必须由网关统一持有和使用上游密钥，不要求下游客户端直接持有真实供应商密钥。

### FR-018 敏感信息脱敏

系统必须在日志和审计中对密钥、Authorization 头和可能的敏感字段进行脱敏。

## 10.5 P0：日志、审计与可观测性

### FR-019 请求日志

系统必须记录请求时间、耗时、入口协议、目标上游协议、模型、状态码和 request id。

### FR-020 错误日志

系统必须记录协议转换失败、上游超时、上游鉴权失败、响应解析失败等关键错误。

### FR-021 Usage 统计

系统必须尽可能统一记录 input tokens、output tokens、总 tokens 和请求计数；若上游字段不完全一致，应以归一化字段输出并保留原始值。

### FR-022 审计查询基础能力

系统应支持按时间、协议、模型、状态码和 request id 查询调用记录。

## 10.6 P0：兼容性与错误处理

### FR-023 参数兼容策略

系统必须定义“不支持字段”“弱兼容字段”“可透传字段”三类策略，并在转换失败时给出明确错误。

### FR-024 错误语义统一

系统必须尽量保持下游协议原生错误风格，例如 Anthropic 风格错误返回 Anthropic 兼容结构，OpenAI 风格错误返回 OpenAI 兼容结构。

### FR-025 能力降级说明

当某些上游能力无法完整映射时，系统必须支持明确报错或按预定义降级规则处理，不能静默丢失关键语义。

## 10.7 P1：治理增强能力

### FR-026 限流

系统应支持按客户端、模型、接口类型进行基础限流。

### FR-027 并发控制

系统应支持全局并发和单上游并发保护，避免本地网关过载。

### FR-028 熔断与重试

系统应支持上游失败熔断、有限重试和短期摘除。

### FR-029 配置热更新

系统应支持在不中断服务的前提下刷新模型映射和部分路由配置。

### FR-030 调试模式

系统应支持受控调试模式，用于输出转换前后的规范化请求摘要和上游响应摘要。

## 11. 关键转换规则分析

## 11.1 消息模型转换

系统需要统一处理以下差异：

- Anthropic 的 `messages + system`
- OpenAI Chat 的 `messages(role=system|user|assistant|tool)`
- OpenAI Responses 的 `instructions + input + output items`

网关内部应收敛为统一消息段模型，至少包含：

- role
- content parts
- tool call
- tool result
- metadata

## 11.2 工具调用转换

系统需要支持：

- Anthropic `tool_use / tool_result`
- OpenAI Chat `tools / tool_calls / tool`
- OpenAI Responses 的工具定义与 output item

重点要求：

- 工具名称保持稳定
- 参数 schema 尽量原样保留
- 工具调用 id 在跨协议转换时可追踪
- 无法等价映射时明确报错

## 11.3 流式事件转换

系统需要处理：

- Anthropic SSE 事件
- OpenAI Chat chunk 流
- OpenAI Responses event 流

转换时至少保证：

- 首事件可被客户端识别
- 增量文本顺序正确
- 工具调用增量可恢复
- 完成事件和终止原因完整
- 中途失败可输出兼容错误事件或中止响应

## 11.4 停止原因与状态映射

系统需要统一处理：

- Anthropic `stop_reason`
- OpenAI `finish_reason`
- Responses `status / incomplete_details`

系统内部应有一套标准化结束原因，例如：

- `completed`
- `max_tokens`
- `tool_use`
- `stop_sequence`
- `content_filtered`
- `upstream_error`

随后再映射回下游协议字段。

## 11.5 Usage 映射

不同协议的 usage 字段命名和粒度可能不同，系统应至少归一化为：

- `input_tokens`
- `output_tokens`
- `total_tokens`
- `cache_creation_tokens`
- `cache_read_tokens`

如果某上游不提供某项数据，可为空，但需要保留原始 usage 片段用于审计。

## 12. 接口边界要求

## 12.1 下游接口边界

建议至少暴露这些入口：

- `POST /anthropic/v1/messages`
- `POST /openai/v1/chat/completions`
- `POST /openai/v1/responses`

是否额外兼容官方默认路径，可在设计阶段确定，但本期需求要求至少支持一套稳定前缀，避免与现有管理服务 API 冲突。

## 12.2 管理与治理接口边界

除协议接口外，建议提供最小治理接口：

- `GET /healthz`
- `GET /readyz`
- `GET /admin/routes`
- `GET /admin/models`
- `GET /admin/requests/{id}`

这些接口不属于下游兼容协议的一部分，但属于网关可运维性要求。

## 13. 非功能需求

## 13.1 性能

- 本地非流式代理的额外处理开销应尽量控制在低毫秒级
- 流式首包延迟不应因转换层被明显放大
- 在单机场景下应支持基础并发访问

## 13.2 稳定性

- 网关进程异常时应输出明确错误日志
- 上游不可用时应快速失败或按回退策略处理
- 配置错误应在启动阶段尽早暴露

## 13.3 可维护性

- 协议适配层应按供应商和协议拆分，避免耦合
- 统一语义模型应成为新增协议的唯一接入面
- 错误码、日志字段和审计字段应标准化

## 13.4 安全性

- 默认只监听 `127.0.0.1`
- 默认不对外网开放
- 配置中的密钥不得明文打印到日志

## 14. 版本优先级建议

## 14.1 P0 首版闭环

首版建议优先交付：

- 本地启动与配置加载
- Anthropic / `chat/completions` / `responses` 三个入口
- 非流式请求转换
- 基础流式透传与转换
- 模型别名与上游路由
- 统一日志、request id、健康检查
- 基础错误处理和 usage 归一化

## 14.2 P1 增强

第二阶段再补：

- 回退策略
- 限流、并发控制、熔断
- 调试接口与请求回放
- 配置热更新
- 更完整的工具调用和流式事件兼容

## 14.3 P2 后续演进

后续可扩展：

- 图片、音频、embedding 等更多接口
- 多租户和细粒度鉴权
- 成本分析报表
- 与管理服务深度整合
- 远程部署与集群模式

## 15. 验收口径

满足以下条件，可认为本地 AI 协议转换网关达到首版可用：

- 客户端可以只改 Base URL，即通过本地网关调用至少一套兼容协议
- 网关可把下游 Anthropic 请求路由到上游 Anthropic 或 OpenAI 协议之一
- 网关可把下游 `chat/completions` 请求路由到上游 `chat/completions` 或 Anthropic
- 网关可把下游 `responses` 请求路由到上游 `responses` 或 Anthropic
- 非流式调用能返回兼容结构
- 流式调用能返回可消费的兼容事件流
- 工具调用在至少一条跨协议链路上可用
- 日志中可追踪 request id、入口协议、目标协议、模型、耗时和结果
- 上游失败时，客户端能收到明确且协议风格兼容的错误

## 16. 风险与难点

- Anthropic、Chat、Responses 三套协议并不是一一等价，尤其是流式事件和工具调用
- `responses` 的对象模型比传统 Chat 更灵活，向旧协议回写时容易丢语义
- 不同供应商的 usage 和停止原因粒度不同，完全对齐成本高
- 官方 SDK 对路径、头部和 SSE 事件格式可能有细节依赖，兼容测试必须充分
- 如果内部不先建立统一语义层，后续协议扩展会迅速失控

## 17. 与现有项目关系

建议把本地 AI 协议转换网关视为 `icoo_gateway` 的“数据面 / 协议面”能力，而现有“网关管理服务”文档更偏“控制面 / 资源面”能力。

两者关系建议如下：

- 协议转换网关负责模型调用入口、协议兼容、上游路由、usage 与错误治理
- 网关管理服务负责资源管理、Agent/Team/Conversation 管理和编排配置
- 后续可由管理服务下发模型路由、别名和权限策略给协议转换网关

## 18. 结论

本地 AI 协议转换网关的本质，是在本机侧建立一个“统一 AI 协议兼容层”。它既要解决“客户端只想保留原协议调用方式”的接入问题，也要解决“平台希望统一管理上游模型和密钥”的治理问题。

从需求优先级看，首版最重要的不是一次性覆盖所有供应商全部细节，而是先建立：

- 稳定的三协议入口
- 可演进的统一语义转换层
- 清晰的模型路由与错误处理机制
- 可追踪的日志和审计能力

只要这四个基础面先立住，后续无论是继续扩协议、接管理面，还是增加回退、限流、热更新，都会容易很多。
