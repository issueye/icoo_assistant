# v0.1.26 开发计划

## 1. 版本主题

让 `task_audit history` 在聚焦失败查看时补上一条 latest failure error hint，减少用户还要下翻条目列表确认最新失败原始错误文案。

## 2. 为什么现在做

`v0.1.25` 已经能让用户更快把最新失败提示对到具体条目，下一步最自然的问题就是“这条最新失败在顶部能不能直接看到原始错误文本”。这一版继续停留在 history 可读性层，不扩展到新的诊断逻辑。

## 3. 计划变更

- 为 `task_audit history` 增加一个轻量 latest failure error hint
- 让失败历史视图更容易快速对齐最新失败提示与原始错误文本
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 自动提炼更长的错误摘要
- 新的错误归因逻辑
- 基于错误文本的修复建议

## 5. 验收标准

- [x] `task_audit history` 新增 latest failure error hint
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.26` 可以判定为已完成，当前任务侧审计已经能在 history 视图里通过 `latest_failure_error` 直接提示最新失败样本的原始错误文本，让用户在看到命令、签名、时间和条目位置前，就能先快速确认这次失败具体报了什么。

完成依据：

- 已在 `task_audit history` 中新增 `latest_failure_error`
- 当前 `status=failed` 与 `reason=<reason>` 的失败聚焦历史查看都会直接提示最新失败对应的原始错误文本与 `job_id`
- README 与 `tool_catalog` 已同步补上 latest failure error 提示说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 `latest_failure_error` 仍然只是原始错误文本提示，不承担错误摘要提炼
- 当前只提示最新失败的 error 文本，不扩展到修复建议
- 当前仍然依赖 history 返回中的最近一条失败记录

## 7. 风险与停手线

- 如果 latest failure error hint 开始承载摘要提炼，会和现有 `latest_failure_signature` 混淆
- 如果错误文本过长且不做边界控制，可能削弱 history 头部的紧凑性
- 因此本版应控制在“更快看到原始错误文本”而不是“解释错误意味着什么”
