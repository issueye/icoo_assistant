# v0.0.5 开发计划

## 1. 版本定位

`v0.0.5` 聚焦把已经存在的项目任务系统和后台执行系统连接起来。核心目标是补齐“项目任务如何关联实际执行”的桥梁，让 Agent 可以围绕任务启动后台工作并追踪执行结果。

## 2. 版本目标

本版本计划交付以下能力：

- 项目任务与后台任务的关联约定
- 启动后台任务时携带 `task_id`
- 查询任务时可以看到关联执行上下文
- 相关测试和文档

## 3. 范围定义

### 3.1 纳入 `v0.0.5`

- `project_task` 与 `background` 的协同约定
- 任务执行关联字段或展示增强
- 文档与测试

### 3.2 暂不纳入 `v0.0.5`

- 自动任务拆解
- 自动认领任务
- 多 Agent 团队协作
- 审批协议
- worktree 绑定

## 4. 设计重点

### 4.1 目标关系

- `todo` 负责当前会话内短期规划
- `project_task` 负责项目级任务状态
- `background` 负责实际长耗时执行
- 三者之间保持清晰边界，但允许通过 `task_id` 建立关联

### 4.2 版本重点

- 先解决“看得见关联”问题
- 再考虑“自动驱动状态变化”问题

## 5. 任务拆解

### A. 关联字段

- 梳理任务与后台执行的最小关联字段
- 统一展示格式

### B. 工具协同

- 让 `background` 工具更清晰地暴露 `task_id`
- 让 `project_task` 输出中体现执行关联信息

### C. 测试与文档

- 增加协同测试
- 更新 README
- 更新版本文档

## 6. 验收清单

- [x] 后台任务可明确关联项目任务 ID
- [x] 项目任务相关输出可体现执行关联
- [x] 测试通过
- [x] 文档更新

## 7. 完成度评估

### 7.1 当前结论

`v0.0.5` 可以判定为已完成，项目任务系统和后台执行系统已经建立起最小可用的关联展示闭环。

完成依据：

- `background` 已支持按 `task_id` 过滤查询
- `background` 启动结果会明确显示关联任务
- `project_task` 查询结果已展示关联后台任务上下文
- 已补充协同测试，`go test ./...` 通过
- README 已同步更新能力说明

### 7.2 当前边界

- 当前仍以“展示关联”为主，尚未自动驱动任务状态变化
- 任务和后台执行之间目前是弱耦合关系，便于后续继续演进
- 关联展示优先可读性，后续如果上下文成本过高仍可继续压缩
## 8. 风险提示

- 如果过早把任务状态自动联动做得太重，后续多 Agent 调度会变复杂
- 展示格式如果过度冗长，会影响上下文成本
- 关联关系需要兼顾可观察性和后续可扩展性
