# icoo_proxy v0.2-v0.4 版本路线图

## 1. 文档目标

本文档用于明确 `icoo_proxy` 从当前开发态走向“可用交付产品”的版本路线，帮助后续开发、测试、验证和交付节奏保持一致。

本文档重点回答三个问题：

- 当前版本已经到哪一步
- 还需要多少个版本才能达到可交付
- 每个版本最应该完成什么

## 2. 当前完成度判断

截至当前版本，`icoo_proxy` 已经具备这些基础能力：

- `Go + Wails v2` 桌面应用骨架
- `Vue 3 + Vue Router + Pinia + JavaScript + TailwindCSS` 前端控制台
- 本地 HTTP AI 网关
- Anthropic、OpenAI `chat/completions`、OpenAI `responses` 三套入口
- 同协议透传
- 非流式跨协议转换
- 基础 function tools 定义与调用映射
- 最近请求摘要
- 供应商管理
- 默认路由策略管理

这说明项目已经不再是纯原型，而是进入了“内部试用前夜”的阶段。

但距离“可用交付产品”仍有明显缺口：

- 跨协议流式转换尚未完成
- 供应商健康检查和连通性验证不足
- 路由策略还停留在默认协议级，未到模型别名级
- 日志、审计、错误排查视图仍偏基础
- 缺少更完整的配置校验、边界提示和安装交付口径

## 3. 产品分级口径

为了避免“可用”定义过于模糊，建议把后续目标分成三档：

## 3.1 内部试用版

满足这些条件即可：

- 桌面端可以正常安装、启动和关闭
- 三套协议在常见文本场景中可稳定调用
- 关键跨协议链路可工作
- 供应商和默认路由策略可管理
- 发生错误时，用户能获得明确提示

## 3.2 首个交付版

除内部试用版条件外，还应满足：

- 流式调用至少覆盖一条关键跨协议链路
- 路由策略具备更细粒度配置
- 供应商健康检查可用
- 请求日志和筛选能力达到可排障水平
- 文档、默认配置、构建交付方式完整

## 3.3 稳定产品版

除首个交付版条件外，还应满足：

- 主要协议能力边界清楚且测试充分
- 复杂工具调用兼容更完整
- 稳定性、恢复性和可观测性达到持续使用水平
- 面向真实团队使用的运维体验更成熟

## 4. 版本数量预估

按当前完成度，我对版本数量的判断如下：

- 再经过 `3` 个版本，可以达到“内部试用版”
- 再经过 `5` 个版本左右，可以达到“首个可交付版”
- 如果目标是“更稳的产品版”，建议按 `6-7` 个版本规划

因此，建议把近期版本目标定为：

- `v0.2.x`：内部试用准备
- `v0.3.x`：试用交付
- `v0.4.x`：首个正式可交付版

## 5. 推荐路线

## v0.2.x：从开发原型走向内部试用准备

### 版本目标

把当前“功能已经有了”的系统，推进到“团队内部愿意开始试”的程度。

### 核心任务

- 完成关键跨协议流式转换
- 增加供应商健康检查
- 增加默认路由策略校验
- 增加更明确的配置错误提示
- 增加请求日志基础筛选能力
- 增加构建和运行的标准化说明

### 交付标准

- 关键常用场景不再只支持非流式
- 供应商配置错误能被快速识别
- 路由策略错误不会静默生效
- 桌面端能承担基本的内部试用入口

## v0.3.x：内部试用交付版

### 版本目标

把产品推进到“可以给内部团队、种子用户试用”的阶段。

### 核心任务

- 路由策略下沉到模型别名级或规则级
- 增加请求日志页、状态筛选和错误聚合
- 增加供应商联通性测试按钮和状态缓存
- 补足更常见工具调用兼容场景
- 增强桌面端表单体验与校验逻辑
- 补齐 README、环境变量说明和最小上手路径

### 交付标准

- 内部团队可通过桌面端完成主要配置
- 出现问题时能靠日志页和状态页定位
- 多供应商、多模型路由行为可解释
- 试用中不再严重依赖开发者手工改 `.env`

## v0.4.x：首个正式可交付版

### 版本目标

形成一个边界清楚、能力稳定、可以对外或对内部业务线正式交付的版本。

### 核心任务

- 继续完善流式跨协议兼容
- 增加更细致的审计视图
- 增加失败重试、限流或最小保护机制
- 完善首次启动与默认配置体验
- 完善安装包、版本号和交付文档
- 基于试用反馈做一轮稳定性收敛

### 交付标准

- 具备明确安装与使用口径
- 关键协议能力经过回归验证
- 主要错误路径可诊断、可恢复
- 产品边界和“未支持能力”已经写清楚

## 6. 不建议过早做的内容

在 `v0.2-v0.4` 这条线上，不建议太早投入这些方向：

- 复杂多租户和权限体系
- 企业级计费与成本中心
- 分布式集群化部署
- 图片、音频等更多模态的大范围扩展
- 过度复杂的工作流画布

原因很简单：这些方向会明显分散当前主线，而当前主线最重要的是先把“本地 AI 网关”做成真正可交付的文本协议产品。

## 7. 当前最合理的下一步

从路线图往回推，当前最合理的近期任务优先级建议如下：

1. 供应商健康检查与连通性状态
2. 默认路由策略的模型别名级细化
3. 跨协议流式转换
4. 请求日志页与筛选能力
5. 更完整的工具调用边界处理

## 8. 结论

`icoo_proxy` 目前已经跨过了“是否能做出来”的阶段，正在进入“如何尽快做成可交付产品”的阶段。

如果按当前开发效率继续推进，最合理的判断是：

- `v0.2.x` 进入内部试用准备
- `v0.3.x` 达到内部试用交付
- `v0.4.x` 达到首个正式可交付版

也就是说，从现在算起，预计再经过 `3 到 5` 个版本迭代，可以把它推进到一个真正可用且可交付的本地 AI 网关产品。
