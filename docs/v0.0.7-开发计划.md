# v0.0.7 开发计划

## 1. 版本定位

`v0.0.7` 聚焦把当前“只保留最近一次后台执行结果”的能力推进到“保留可追溯执行历史”。核心目标是为项目任务补齐基础执行审计信息，让多次运行、失败重试和后续排查有据可查。

## 2. 版本目标

本版本计划交付以下能力：

- 项目任务的后台执行历史记录
- 基础历史展示能力
- 相关测试和文档

## 3. 范围定义

### 3.1 纳入 `v0.0.7`

- 项目任务执行历史字段或存储结构
- 查询任务时展示历史摘要
- 文档与测试

### 3.2 暂不纳入 `v0.0.7`

- 自动任务拆解
- 自动认领任务
- 多 Agent 团队协作
- 审批协议
- worktree 绑定

## 4. 设计重点

### 4.1 核心原则

- 保持存储结构简单
- 历史记录优先服务排查和追踪
- 避免让单个任务对象无限膨胀

### 4.2 版本重点

- 先保留有限历史摘要
- 再考虑完整事件流或独立审计日志

## 5. 任务拆解

### A. 数据结构

- 设计任务执行历史结构
- 约束保留条数或摘要大小

### B. 展示输出

- 在 `project_task` 中展示最近历史
- 与当前 `last_background` 信息协调

### C. 测试与文档

- 增加历史记录测试
- 更新 README
- 更新版本文档

## 6. 验收清单

- [x] 项目任务可保留多次后台执行历史
- [x] 查询任务时可看到历史摘要
- [x] 测试通过
- [x] 文档更新

## 7. 完成度评估

### 7.1 当前结论

`v0.0.7` 可以判定为已完成，项目任务现在已经具备有限执行历史记录和紧凑历史展示能力。

完成依据：

- 已为任务新增后台执行历史字段
- 历史记录已做上限控制，避免无限增长
- `project_task` 查询结果可展示历史摘要
- 已补充历史记录测试，`go test ./...` 通过
- README 已同步更新能力说明

### 7.2 当前边界

- 当前只保留有限条数的摘要历史，不是完整审计日志
- 历史仍然内嵌在任务对象里，尚未拆到独立存储
- 展示重点仍然是最近记录，深度检索能力留给后续版本
## 8. 风险提示

- 历史记录如果无限增长，会抬高上下文和存储成本
- 展示方式需要兼顾可读性和紧凑性
- 历史结构设计要为后续独立审计日志留出口
