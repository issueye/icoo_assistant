# v0.1.30 开发计划

## 1. 版本主题

收口初代可用版本交付，补齐最终上手说明、验证清单和当前限制说明，让仓库达到“新用户按 README 能完成一次最小闭环”的可交付状态。

## 2. 为什么现在做

`v0.1.27` 到 `v0.1.29` 已经依次补齐了环境自检、最小 happy path 和常见降级场景反馈。到 `v0.1.30`，重点不应再是新增零散能力点，而是把这些能力收成一个真正可交付的初代版本说明与验证口径。

## 3. 计划变更

- 收口 README 中的上手、排障、边界和当前限制说明
- 补齐面向 `v0.1.30` 的最终验证清单
- 明确 fake 模式与真实模型模式在当前版本中的边界
- 对照路线图完成初代可用版交付评估

## 4. 本版不做

- 安装器或桌面封装
- 多模型适配扩展
- 复杂自动修复与联网诊断

## 5. 验收标准

- [x] README 已能支撑新用户完成一次最小闭环
- [x] 当前限制与模式边界说明已清楚写明
- [x] 面向交付的验证清单已补齐
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.30` 可以判定为已完成，当前仓库已经具备一个边界清楚、验证口径明确的初代可用版本：新用户可以先执行 `assistant check`，再按 README 的固定最小 happy path 继续验证工具边界、任务入口和审计入口；同时 fake 模式与真实模式的边界、当前限制和交付清单也已经明确写清。

完成依据：

- README 已收口最小 happy path、常见卡点、模式边界和当前限制
- CLI `--help` 已补充首次使用与 fake/real 模式说明
- `v0.1.30` 验证清单已补齐
- 已完成 `go test ./...` 与关键命令 smoke 验证

### 6.2 当前边界

- 当前版本仍以本地 CLI 交付为主，不包含安装器或桌面封装
- fake 模式只适用于 dry run 和路径验证，不等价于真实模型体验
- 当前仍然聚焦最小闭环，不追求复杂生产工作流

## 7. 风险与停手线

- 如果继续追加新的局部功能点，会打散 `v0.1.30` 的交付收口
- 如果只宣称“可用”但没有验证口径，最终版本仍然难以评估
- 因此本版应控制在“交付收口与验证清单”而不是“再开新主题”
