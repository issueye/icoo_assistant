# v0.1.20 开发计划

## 1. 版本主题

让 `task_audit history` 在失败条目里直接展示归类后的 failure reason，减少用户在 history 与 summary 之间来回对照。

## 2. 为什么现在做

`v0.1.19` 已经让 focused history 能看出 latest / previous 以及两条样本的主要差别，下一步最自然的问题就是“这条失败到底属于哪一类原因”。这一版继续停留在 history 可读性层，不扩展到新的诊断逻辑。

## 3. 计划变更

- 为失败历史条目增加直接可见的 failure reason
- 让 `status=failed` 或 `reason=<reason>` 的 history 视图更容易快速扫读
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 新的失败原因分类体系
- 多条历史的聚类视图
- 自动原因解释

## 5. 验收标准

- [x] `task_audit history` 的失败条目新增 reason 展示
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.20` 可以判定为已完成，当前任务侧审计已经能在 history 的失败条目里直接展示归类后的 `reason=`，让用户在查看具体历史时不用再频繁回到 summary 对照失败原因。

完成依据：

- 已在 `task_audit history` 的失败条目中新增 `reason=<reason>`
- 当前 `status=failed` 与 `reason=<reason>` 的历史查看都会直接显示原因标签
- README 与 `tool_catalog` 已同步补上 history 原因标签说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 reason 展示仍然沿用现有轻量分类体系
- 当前只在失败条目中展示 reason，不影响成功条目输出
- 当前不会在 history 条目里继续展开原因解释

## 7. 风险与停手线

- 如果 history 条目塞入过多字段，会削弱当前紧凑性
- 如果 reason 展示开始承载解释语义，会和 summary 的失败分类字段重叠
- 因此本版应控制在“更快看懂条目”而不是“追加新的判断层”
