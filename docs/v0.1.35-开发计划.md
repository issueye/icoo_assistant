# v0.1.35 开发计划

## 1. 版本主题

在 Team registry 和 inbox message store 已落地之后，补上最小的 lead -> teammate -> lead 通信闭环：request、reply 和 thread 查询。

## 2. 为什么现在做

`v0.1.33` 已经完成了 Team roster，`v0.1.34` 已经完成了 `.team/inbox/` message store。但如果只停在“能写消息、能看 inbox”，还不能说明 Team 主线已经具备最小通信能力，因为请求和响应之间还没有被明确关联起来。

根据路线图，`v0.1.35` 应该打通 lead -> teammate 的最小发送与响应闭环。当前还不适合直接做真正的 teammate loop 或自动执行，因为在那之前，更基础的问题是：请求是否有 ID、回复是否能自动回到 lead、整条线程能否被回看。

因此，本版只回答一个问题：如何让一条最小 Team 请求具备“可发起、可响应、可追踪”的基本结构。

## 3. 计划变更

- 在 `internal/team` 中增加 `ReplyToRequest / ListThread` 能力
- 在 `team_message` 工具中增加 `request / reply / thread`
- 让 `reply` 能基于 `request_id` 自动回复到原请求发起者
- 更新 README 与工具边界说明，明确当前已经具备最小通信闭环
- 补充对应测试

## 4. 本版不做

- 真正运行中的 teammate loop
- 协议审批、自动认领和身份重注入
- 复杂消息状态机或已读确认系统

## 5. 验收标准

- [x] `team_message action=request` 可创建带 `request_id` 的请求
- [x] `team_message action=reply` 可按 `request_id` 自动回复到原请求发起者
- [x] `team_message action=thread` 可查看完整 request/response 线程
- [x] README 已补充最小通信闭环示例
- [x] `go test ./...` 通过

## 6. 完成度评估

`v0.1.35` 可以判定为已完成。本版把 Team 主线从“只有 inbox store”推进到了“最小请求/响应闭环”，但仍然刻意停在手动驱动的 request/reply/thread，而不是直接跳到真正运行中的 teammate loop。

本版完成情况如下：

- `internal/team` 新增 `ReplyToRequest` 和 `ListThread`
- `team_message` 工具已支持 `request / reply / thread`
- 回复会按 `request_id` 找到原请求，并自动把响应回送给原发起者
- README 已新增最小通信闭环示例命令
- 相关测试已补齐并通过

当前边界判断：

- 这版已经满足 Stage B 第三小步“最小 lead -> teammate 发送与响应闭环”
- 还没有进入真正运行中的消息消费或 Agent loop
- 下一版如果继续 Team 主线，应开始评估协议化 request/response 或 teammate loop 的承载方式，而不是继续只补 message 字段

## 7. 风险与停手线

- 如果继续只在 `request_id`、消息字段上追加细节，会再次退回字段级机械迭代
- 如果现在直接接入复杂协议，会跳过“谁来消费 inbox、谁来产生响应”这个更直接的问题
- 因此本版停在“请求可发起、响应可回送、线程可查看”，把下一步空间留给后续协议或 teammate loop 版本
