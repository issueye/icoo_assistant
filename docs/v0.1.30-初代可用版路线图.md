# v0.1.30 初代可用版路线图

## 1. 交付目标

`v0.1.30` 不是再补一个局部字段，而是要交付一个“新用户按 README 操作后，能够实际跑通一次最小使用闭环”的初代可用版本。

这里的“可以使用”至少包含：

- 能完成基础启动与版本/帮助查看
- 能在 fake client 或真实 API Key 场景下跑通一次单轮任务
- 能创建项目任务，并关联一次后台执行或审计查看
- 能通过 README 找到清晰的上手顺序、关键边界和排障入口
- 遇到明显的配置或运行前提问题时，CLI 能给出明确反馈

## 2. 当前判断

截至 `v0.1.26`，项目已经具备了不错的内部骨架与审计能力，但“对新用户是否好上手、是否能快速判断自己能不能跑起来、是否有稳定的最小闭环”还没有被当作主线交付目标来推进。

这意味着从 `v0.1.27` 到 `v0.1.30`，版本重点需要从字段级可读性优化切到“可用性闭环”。

## 3. 倒排节奏

### v0.1.27

目标：补齐环境自检入口，先解决启动前判断问题。

建议交付：

- CLI 自检命令
- 关键配置与工作区前置检查
- README 自检与上手说明

### v0.1.28

目标：补齐最小 happy path，让用户能按固定顺序跑通一次真实可感知的使用流程。

建议交付：

- 一条清晰的最小使用路径
- 更明确的示例命令顺序
- 对任务、后台执行、审计入口的串联说明

### v0.1.29

目标：补齐可用版的稳定性门槛，让常见失败场景不会把用户直接卡死。

建议交付：

- 更明确的错误提示或降级反馈
- 最小闭环的 smoke 验证
- README 排障说明收口

### v0.1.30

目标：交付初代可用版本并收口文档、验证和边界说明。

建议交付：

- 可执行的最小闭环已稳定可跑
- README 已能支撑新用户上手
- 关键命令、边界和限制已清楚写明
- 至少一套面向交付的验证清单

## 4. v0.1.30 的验收口径

到 `v0.1.30` 时，建议至少满足这些检查项：

- [ ] `go run ./cmd/assistant --version` 与 `--help` 正常
- [ ] 用户可以执行一次环境自检并得到明确结果
- [ ] 用户可以按 README 跑通一次最小任务闭环
- [ ] 用户可以定位项目任务、后台执行和审计入口的职责边界
- [ ] `go test ./...` 通过
- [ ] README 已补齐上手、排障、边界和当前限制

## 5. 当前策略调整

- 从 `v0.1.27` 起，`task_audit history` 的字段级微调不再作为默认主线
- 新版本优先服务 `v0.1.30` 的可用交付，不再继续机械地沿同一条小字段链路推进
- 如果某个小改动不能直接提高“可启动、可跑通、可理解、可排障”中的至少一项，就不应优先进入 `v0.1.27` 到 `v0.1.30`
