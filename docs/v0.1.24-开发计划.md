# v0.1.24 开发计划

## 1. 版本主题

让 `task_audit history` 在聚焦失败查看时补上一条 latest failure updated_at 提示，减少用户继续在条目里手动确认这次失败的时间点。

## 2. 为什么现在做

`v0.1.23` 已经能让用户更快看到最新失败的错误模式，下一步最自然的问题就是“这条最新失败具体发生在什么时候”。这一版继续停留在 history 可读性层，不扩展到新的推断逻辑。

## 3. 计划变更

- 为 `task_audit history` 增加一个轻量 latest failure updated_at 提示
- 让失败历史视图更容易快速对齐当前失败记录与时间点
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 时间窗口分析
- 自动时间趋势结论
- 更复杂的失败时间聚合

## 5. 验收标准

- [x] `task_audit history` 新增 latest failure updated_at 提示
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.24` 可以判定为已完成，当前任务侧审计已经能在 history 视图里通过 `latest_failure_updated_at` 直接提示最新失败样本的时间点，让用户在定位到最新失败、命令和签名后，也能立刻对齐发生时间。

完成依据：

- 已在 `task_audit history` 中新增 `latest_failure_updated_at`
- 当前 `status=failed` 与 `reason=<reason>` 的失败聚焦历史查看都会直接提示最新失败时间
- README 与 `tool_catalog` 已同步补上 latest failure updated_at 提示说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 `latest_failure_updated_at` 仍然只是轻量提示，不承担时间趋势解释
- 当前只提示最新失败时间，不扩展到时间窗口分析
- 当前仍然依赖 history 返回中的最近一条失败记录

## 7. 风险与停手线

- 如果 latest failure updated_at 提示与现有 latest_sample 重复过多，会削弱 history 的紧凑性
- 如果提示开始承载时间趋势解释，会和 summary 的趋势字段混在一起
- 因此本版应控制在“更快看到时间点”而不是“解释时间变化意味着什么”
