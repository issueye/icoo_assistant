# v0.0.1 开发计划

## 1. 版本定位

`v0.0.1` 定位为项目首个可演示、可测试、可继续迭代的单 Agent MVP。目标不是一次性做完教程里的全部能力，而是先把“本地可运行的编码 Agent 闭环”打磨稳定。

## 2. 版本目标

本版本需要满足以下最小交付标准：

- 能通过 CLI 启动并执行单轮或 REPL 交互
- 能完成文件读取、写入、编辑与命令执行
- 能维护 Todo 状态
- 能在上下文过长时进行压缩并写出 transcript
- 能加载本地 Skill
- 能通过 `task` 工具委托 Subagent 并回收摘要
- 能在当前平台稳定运行基础命令

## 3. 范围定义

### 3.1 纳入 `v0.0.1`

- `cmd/assistant` CLI 入口
- `internal/agent` 对话与工具循环
- `internal/tools` 的基础工具集
- `internal/workspace` 工作区路径沙箱
- `internal/todo` 会话级 Todo 管理
- `internal/compact` 压缩与 transcript
- `internal/skill` 技能加载
- `internal/subagent` 子代理摘要执行
- 基础单元测试
- 最小使用文档与版本信息

### 3.2 暂不纳入 `v0.0.1`

- 持久化任务系统
- 后台任务
- 多 Agent 团队
- 协议审批
- 自动认领任务
- worktree 隔离执行
- Web 控制台

## 4. 风险与收敛策略

### 风险 1：平台兼容性

当前最关键的发布风险是命令执行能力。如果命令工具只依赖单一 shell，会直接影响 MVP 可用性。

收敛策略：

- 优先保证当前平台可执行命令
- 将命令工具做成跨平台默认实现
- 用自动化测试覆盖 Windows 和非 Windows 的基础命令路径

### 风险 2：版本边界膨胀

教程覆盖范围很大，容易把 `v0.0.1` 拉成“大而全”。

收敛策略：

- `v0.0.1` 只对“单 Agent 本地闭环”负责
- P1/P2 能力全部延后到后续版本

## 7. 完成度评估

### 7.1 当前结论

`v0.0.1` 可以判定为已完成，适合作为首个可演示的单 Agent MVP 基线版本。

完成依据：

- `go test ./...` 已通过
- `go run ./cmd/assistant --version` 输出 `0.0.1`
- `go run ./cmd/assistant --help` 可展示使用说明
- 当前平台下命令工具已支持稳定执行基础命令
- README、版本计划和配置模板已经补齐

### 7.2 已知剩余风险

- 真实 Anthropic API 的联调仍依赖本地 `ANTHROPIC_API_KEY` 和实际模型权限，当前仓库内未做在线验收
- `v0.0.1` 仍然是单 Agent MVP，不包含任务持久化、后台任务和多 Agent 能力
- CLI 仍是轻量入口，尚未引入更完整的子命令结构
## 8. 后续版本建议

- `v0.0.2`：任务持久化与项目级任务管理
- `v0.0.3`：后台任务与更完整的 Skill/Subagent 协作
- `v0.1.0`：多 Agent、协议、worktree 隔离
