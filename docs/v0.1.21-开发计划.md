# v0.1.21 开发计划

## 1. 版本主题

让 `task_audit history` 在聚焦失败查看时补上更直接的 latest sample 提示，减少用户自己判断当前最值得先看的那条记录。

## 2. 为什么现在做

`v0.1.20` 已经让 history 条目里能直接看到 failure reason，下一步最自然的问题就是“当前应该先盯哪条最新失败”。这一版继续停留在历史可读性层，不扩展到新的分析逻辑。

## 3. 计划变更

- 为 `task_audit history` 增加一个轻量 latest sample 提示
- 让失败历史视图更容易快速定位当前最新失败记录
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 自动推荐修复动作
- 新的失败原因分类
- 多任务联合视图

## 5. 验收标准

- [x] `task_audit history` 新增 latest sample 提示
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.21` 可以判定为已完成，当前任务侧审计已经能在 history 视图里通过 `latest_sample` 直接提示当前最新的失败记录，进一步减少用户在多条历史条目之间自己定位重点样本的成本。

完成依据：

- 已在 `task_audit history` 中新增 `latest_sample`
- 当前 `status=failed` 与 `reason=<reason>` 的失败聚焦历史查看都会直接提示最新样本
- README 与 `tool_catalog` 已同步补上 latest sample 提示说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 `latest_sample` 仍然只是轻量定位提示，不承担解释逻辑
- 当前提示仍然依赖返回结果中的最近一条记录，不扩展到跨分页或更长窗口
- 当前不会替代条目本身的阅读，只是帮助更快落点

## 7. 风险与停手线

- 如果 latest sample 提示重复已有条目内容过多，会削弱 history 的紧凑性
- 如果提示开始承载解释语义，会和 summary 的导航字段重叠
- 因此本版应控制在“更快定位最新记录”而不是“追加新的判断层”
