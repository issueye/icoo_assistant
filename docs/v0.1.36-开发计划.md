# v0.1.36 开发计划

## 1. 版本主题

在 `v0.1.35` 最小 request / reply / thread 闭环的基础上，为 Team request 增加可持久化、可查询的协议状态。

## 2. 为什么现在做

`v0.1.35` 已经证明 Team 主线具备了最小通信样本，但当前 request 仍然主要体现为 inbox 里的消息文本。按照路线图，`v0.1.36` 应该开始进入 Protocol 主线，而最稳妥的第一步不是直接做审批或自治，而是先让 request 成为一个可独立查询的对象。

因此，本版只回答一个问题：如何让一次 Team request 不只是“有一条消息线程”，而是“有一个带生命周期的 request record”。

## 3. 计划变更

- 在 `internal/team` 中增加 `.team/requests/` request record 持久化
- 让 `team_message action=request` 自动创建 request record
- 让 `team_message action=reply` 自动把 request 状态更新为 `responded`
- 新增 `team_protocol` 工具用于 `get / list / summary`
- 更新 `assistant check`、README 和工具目录说明
- 补充对应测试

## 4. 本版不做

- 审批流
- auto claim / idle polling
- 真正运行中的 teammate loop
- 多阶段复杂状态机

## 5. 验收标准

- [x] `.team/requests/` 会自动初始化
- [x] request record 会在 `team_message action=request` 时落盘
- [x] request record 会在 `team_message action=reply` 后更新为 `responded`
- [x] 已提供 `team_protocol action=get|list|summary`
- [x] `assistant check` 会显示 request store 已就绪
- [x] README 已更新 `v0.1.36` 使用口径
- [x] `go test ./...` 通过

## 6. 完成度评估

`v0.1.36` 可以判定为已完成。本版把 Team 主线从“只有 request/reply 线程”推进到了“请求对象拥有最小生命周期”，正式进入了 Stage C 的第一步，但仍然刻意停在最小协议层，不提前混入审批、认领和自治循环。

本版完成情况如下：

- `internal/team` 新增 request record 持久化与状态过滤
- `team_message` 的 request / reply 已自动维护协议状态
- `team_protocol` 已支持查看单条 request、筛选 request 和汇总状态
- `assistant check` 已显示 `.team/requests/` 目录
- README 已补充协议层能力说明与示例命令
- 相关测试已补齐并通过

当前边界判断：

- 这版已经满足 Stage C 第一小步“定义并落地 request / response 基础协议”的最小交付
- 还没有进入审批流、auto claim 或真正运行中的 teammate loop
- 下一版应继续围绕 Protocol 主线推进，例如审批入口或认领机制，而不是再次退回 message 字段微调
