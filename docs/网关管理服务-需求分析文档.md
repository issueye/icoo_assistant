# 网关管理服务需求分析文档

## 1. 文档目的

本文档用于定义 `icoo_assistant` 下一阶段“网关管理服务”建设需求，明确该服务需要承接的业务范围、目标边界、核心流程、功能需求与验收口径，为后续服务端开发和版本拆解提供统一基线。

当前仓库中的本地 CLI / Runtime 已经具备单 Agent 执行、工具调用、任务持久化、后台执行与审计能力；此前尝试内置的 Team 相关能力已经移除。后续需要通过独立的网关管理服务，重新承接以下能力：

- Skill 管理
- Agent 管理
- Team 管理
- 单对话管理
- Team 对话管理

## 2. 建设背景

## 2.1 当前现状

当前项目已经完成：

- 单 Agent 本地运行闭环
- 本地工具注册与执行
- 本地 Skill 加载
- 项目任务与后台执行
- 运行审计与基础可观测性

当前项目尚未完成，且不再适合继续内嵌到 CLI Runtime 的能力：

- Team 生命周期管理
- 多 Agent 协同编排
- Team 会话路由
- 持久化的 Agent / Team 管理视图
- 面向未来 Web/控制台的统一管理入口

## 2.2 当前问题

如果继续把 Skill、Agent、Team、对话等治理能力写在本地 Runtime 内，会带来这些问题：

- 本地执行逻辑和平台治理逻辑耦合，版本越来越难维护
- Team / 会话状态难以被统一查看、审计和恢复
- 后续 Web 管理端、外部服务或多终端接入没有统一入口
- 单对话和 Team 对话会演化出两套不一致的数据模型
- Skill、Agent、Team 的绑定关系无法形成平台级配置中心

## 2.3 建设目标

建设一个独立的“网关管理服务”，作为 `icoo_assistant` 的控制面与接入层，负责统一管理 Skill、Agent、Team 与 Conversation 资源，并向 CLI、未来 Web 控制台及外部系统提供稳定 API。

## 3. 产品定位

网关管理服务不是新的本地执行器，也不是替代 `cmd/assistant` 的聊天入口，而是：

- 平台级资源管理中心
- 对话路由与编排入口
- Agent / Team 生命周期管理中心
- Skill 装配与权限边界控制层
- 单对话与 Team 对话的统一接入网关

对应边界如下：

- 本地 Runtime 负责：实际模型调用、工具执行、工作区内落地执行
- 网关管理服务负责：资源治理、配置编排、会话管理、路由分发、状态聚合、审计入口

## 4. 目标用户

### 4.1 直接用户

- 使用本地 `assistant` 的开发者
- 需要统一管理多个 Agent / Team 的研发人员
- 需要通过后台或管理端观察会话与执行状态的项目负责人

### 4.2 间接用户

- 后续 Web 控制台
- 自动化调度服务
- 第三方接入方或企业内部平台

## 5. 本期范围

## 5.1 纳入范围

- Skill 注册、查询、版本管理、启停和绑定
- Agent 模板与实例管理
- Team 定义、成员编排、角色管理
- 单对话创建、续聊、历史查询
- Team 对话创建、消息路由、参与者上下文管理
- 统一资源 ID、状态模型、审计日志模型
- 面向 CLI 和后续控制台的 API

## 5.2 暂不纳入范围

- 复杂权限中心对接
- 企业级多租户计费
- 完整分布式调度系统
- 跨机房高可用架构
- 复杂工作流画布
- 完整 worktree 隔离平台

## 6. 核心业务对象

服务至少需要统一管理以下对象：

- `skill`：可被 Agent 或 Team 使用的能力单元
- `agent_profile`：Agent 模板，定义模型、提示词、工具策略、默认 Skill 等
- `agent_instance`：运行中的 Agent 实例，包含状态、心跳、绑定 Team 等信息
- `team`：一组 Agent 的协作定义
- `team_member`：Team 成员及其角色
- `conversation`：会话主对象，支持单对话和 Team 对话两种模式
- `conversation_message`：会话消息
- `conversation_participant`：会话参与者
- `run`：一次会话驱动或一次编排执行
- `audit_event`：统一审计事件

## 7. 核心场景

## 7.1 Skill 管理场景

1. 管理员注册一个 Skill。
2. 服务保存 Skill 元数据、版本、来源与启用状态。
3. Agent 模板或 Team 可以绑定该 Skill。
4. 对话开始时，服务按配置把可用 Skill 清单下发给目标 Agent。

## 7.2 Agent 管理场景

1. 用户创建一个 Agent 模板。
2. 服务保存模型配置、系统提示词、工具白名单与默认 Skill。
3. 用户基于模板拉起 Agent 实例。
4. 服务跟踪实例状态、最近心跳、所属 Team 和最近会话。

## 7.3 Team 管理场景

1. 用户创建 Team。
2. 向 Team 中添加多个 Agent，并指定 `lead / planner / worker / reviewer` 等角色。
3. 服务保存 Team 拓扑、协作策略与默认对话模式。
4. 后续 Team 对话会基于这套配置进行路由和编排。

## 7.4 单对话场景

1. 用户指定一个 Agent 发起单对话。
2. 服务创建 `conversation(mode=single)`。
3. 消息被路由到目标 Agent。
4. 服务保存消息、运行记录和结果摘要。
5. 用户可以继续追问并查看历史。

## 7.5 Team 对话场景

1. 用户指定一个 Team 发起 Team 对话。
2. 服务创建 `conversation(mode=team)`。
3. 首条消息先路由给 `lead` 或默认入口 Agent。
4. Team 内部可转发子消息、分派任务或回收结果。
5. 服务统一保存外部消息、内部路由、阶段性结论和最终回复。

## 8. 功能需求

## 8.1 P0：平台基础能力

### FR-001 统一资源模型

系统必须为 Skill、Agent、Team、Conversation 提供统一的资源 ID、状态字段、创建时间、更新时间和审计字段。

### FR-002 统一 API 接入

系统必须通过统一 API 暴露 Skill、Agent、Team、Conversation 相关能力，避免 CLI、Web 和后续服务各自维护独立接口。

### FR-003 审计事件记录

系统必须记录资源创建、修改、启停、会话路由、执行失败等关键事件，并支持查询。

### FR-004 配置与实例分离

系统必须区分配置对象与运行对象，例如 `agent_profile` 与 `agent_instance`、`team` 与 `conversation`。

## 8.2 P0：Skill 管理

### FR-005 Skill 注册

系统必须支持创建 Skill，至少包含名称、版本、描述、来源、内容摘要、启用状态。

### FR-006 Skill 查询与列表

系统必须支持按名称、状态、版本查询 Skill，并支持列表分页。

### FR-007 Skill 绑定关系

系统必须支持将 Skill 绑定到 Agent 模板和 Team。

### FR-008 Skill 版本演进

系统必须支持同名 Skill 的版本化管理，并能够标记默认版本。

## 8.3 P0：Agent 管理

### FR-009 Agent 模板管理

系统必须支持 Agent 模板的创建、查询、更新、停用，至少包含模型配置、系统提示词、默认 Skill、工具策略。

### FR-010 Agent 实例管理

系统必须支持基于模板创建 Agent 实例，并记录实例状态，如 `created / idle / busy / offline / disabled`。

### FR-011 Agent 心跳与可用性

运行中的 Agent 实例应支持心跳上报，服务可据此判断在线状态。

### FR-012 Agent 资源视图

系统必须能查看某个 Agent 关联的模板、Skill、Team、最近会话与最近运行状态。

## 8.4 P0：Team 管理

### FR-013 Team 创建与维护

系统必须支持 Team 的创建、查询、更新、停用。

### FR-014 Team 成员管理

系统必须支持向 Team 添加或移除 Agent，并记录成员角色、顺序、启用状态和职责说明。

### FR-015 Team 入口 Agent

系统必须支持定义 Team 默认入口 Agent，作为 Team 对话的第一跳处理者。

### FR-016 Team 协作策略

系统应支持保存 Team 的基础协作策略，如串行分派、主从协作、汇总回复。

## 8.5 P0：单对话管理

### FR-017 单对话创建

系统必须支持以指定 Agent 创建单对话，并返回会话 ID。

### FR-018 单对话续聊

系统必须支持基于会话 ID 续聊，并保留完整消息历史。

### FR-019 会话历史查询

系统必须支持按会话查询消息列表、最近运行结果、参与者与状态。

### FR-020 对话状态管理

系统必须支持会话状态，如 `created / running / waiting / completed / failed / archived`。

## 8.6 P0：Team 对话管理

### FR-021 Team 对话创建

系统必须支持以指定 Team 创建 Team 对话，并返回会话 ID。

### FR-022 Team 内部消息路由

系统必须支持 Team 对话中的消息从外部入口路由到某个成员，再在成员之间转发。

### FR-023 Team 对话视图

系统必须支持区分“对外消息流”和“Team 内部消息流”，避免内部路由污染用户视图。

### FR-024 Team 阶段结论

系统应支持保存 Team 对话过程中的阶段性结论、分工信息和最终汇总结果。

## 8.7 P1：运维与治理能力

### FR-025 运行统计

系统应支持聚合 Agent、Team、Conversation 的基础统计，如会话数、成功率、失败率、平均耗时。

### FR-026 资源启停控制

系统应支持停用 Skill、Agent 模板、Agent 实例和 Team，并阻止其继续接收新请求。

### FR-027 失败重试与补偿

系统应对消息路由失败、Agent 不在线、Team 无入口 Agent 等常见场景提供明确失败原因和可选补偿入口。

### FR-028 外部管理端接入

系统应为后续 Web 控制台预留标准化查询与操作接口。

## 9. 非功能需求

### NFR-001 可扩展性

后续新增模型提供商、Skill 来源类型、Team 协作模式时，不应重构核心数据模型。

### NFR-002 可观测性

服务必须提供结构化日志、审计事件和基础运行指标。

### NFR-003 一致性

会话、Team 路由与资源绑定关系必须可追踪，不能出现“消息存在但参与者丢失”的孤儿数据。

### NFR-004 可恢复性

服务重启后必须可恢复 Skill、Agent、Team、Conversation 元数据与历史记录。

### NFR-005 性能

- 普通管理类查询应控制在秒级内
- 单对话创建与续聊接口应保持低延迟响应
- Team 对话内部路由不得阻塞整个服务进程

### NFR-006 安全性

- 必须具备基础鉴权能力
- 必须避免未授权修改 Agent / Team / Skill 配置
- 必须对内部路由与外部用户视图做隔离

## 10. 约束与边界

当前阶段需要遵守这些约束：

- 不把 Team 管理能力重新写回本地 CLI Runtime
- 不要求第一版就做复杂自治、多租户和审批流
- 单对话和 Team 对话必须复用统一 Conversation 模型
- Skill、Agent、Team 必须先形成管理面，再推进更复杂的协议能力

## 11. MVP 建议范围

建议第一阶段只交付以下最小闭环：

- Skill 基础 CRUD 与版本字段
- Agent 模板 CRUD 与实例状态字段
- Team CRUD 与成员管理
- 单对话创建、续聊、历史查询
- Team 对话创建、入口路由、基础内部转发
- 审计事件记录

以下内容建议放到第二阶段：

- 复杂 Team 协作协议
- 自动认领任务
- 高级调度策略
- 可视化控制台

## 12. 验收标准

当满足以下条件时，可以认为网关管理服务第一阶段需求达标：

- 能创建并查询 Skill、Agent 模板、Agent 实例和 Team
- 能把 Skill 绑定到 Agent 或 Team
- 能发起并续聊单对话
- 能发起 Team 对话，并看到外部消息与内部路由记录
- 能查看 Agent / Team / Conversation 的基本状态
- 能查询关键审计事件
- 本地 Runtime 与网关服务的职责边界清晰，不再依赖 `.team/` 本地目录实现 Team 治理

## 13. 一句话结论

网关管理服务的本质，是把此前散落在本地 Runtime 中不适合继续内嵌的治理、编排和会话路由能力，提升为一个统一、可扩展、可审计的管理控制面。
