# v0.1.31+ 推荐版本路线图

## 1. 文档目的

本文档用于把《当前项目功能与完成度评估报告》与《后续开发计划指南》的结论，进一步收敛成一份可直接指导后续小版本迭代的路线图，避免版本主题再次退回到机械微调或临时发散。

## 2. 当前路线判断

截至 `v0.1.30`，项目已经完成“单 Agent 本地可用版”这一阶段性交付，后续路线应明确分成两段：

- 先完成 `v0.1.31-v0.1.32` 的 post-release hardening
- 再从 `v0.1.33` 起正式进入 Team / Protocol / Worktree 主线

这意味着后续版本不应继续默认围绕零散字段、提示词或 README 局部修饰展开，而应围绕“降低当前可用版真实摩擦”或“推进下一阶段基础设施”来选题。

## 3. 版本推进总规则

- 每个版本只解决一个核心问题
- 每个版本都要有可以检查的产物、入口或测试结果
- 只有当前阶段退出条件基本满足后，才进入下一阶段
- 如果一个版本主题不能明显降低用户摩擦，也不能推进下一主线，就不应进入计划

## 4. Stage A：Post-release Hardening

推荐版本区间：

- `v0.1.31-v0.1.32`

阶段目标：

- 收口 `v0.1.30` 初代可用版的真实上手摩擦
- 修正 CLI、README、自检和最小 happy path 中仍可能误导新用户的点
- 为进入多 Agent 主线建立更稳定的起跳基线

进入条件：

- 当前版本已经是 `v0.1.30` 可用版基线
- 主问题仍集中在上手路径、帮助信息、自检反馈或模式边界

退出条件：

- `assistant check` 的关键信息足够清楚
- `--help` 与 README 的入口一致
- fake / anthropic 两种模式的边界不再容易误解
- 最小 happy path 已能支撑一次稳定上手演示

推荐版本主题示例：

- `v0.1.31`：修正一次真实上手路径中的关键摩擦点，并同步 README 与 smoke 验证
- `v0.1.32`：补齐 CLI 帮助、自检输出或错误提示中的最后一处高频误解

不推荐的版本主题：

- 再补一个没有真实使用价值的审计字段
- 再改一轮没有对应问题反馈的 README 文案
- 在本阶段混入 Team、Protocol 的半成品预研实现

## 5. Stage B：Multi-Agent 基础设施

推荐版本区间：

- `v0.1.33-v0.1.35`

阶段目标：

- 建立 `.team/` 基础结构
- 建立 teammate registry 与最小生命周期
- 建立 lead 到 teammate 的最小消息收发链路

进入条件：

- Stage A 的主要上手摩擦已经收口
- 当前单 Agent 主循环、任务系统和审计入口足够稳定

退出条件：

- 至少支持定义两个 teammate
- lead 可以向指定 teammate 发送消息
- teammate 可以被查询、列出并返回基础状态
- 至少一条最小消息链路可以被审计或复盘

推荐版本主题示例：

- `v0.1.33`：落地 team config 与 teammate registry
- `v0.1.34`：落地 inbox / message store 与基础查询入口
- `v0.1.35`：落地 lead -> teammate 最小发送与响应闭环

不推荐的版本主题：

- 只做“团队概念文档”但没有真实运行入口
- 只补 teammate 字段但没有生命周期或消息能力
- 在消息链路还没通之前先做复杂审批流

## 6. Stage C：Protocol 与自治

推荐版本区间：

- `v0.1.36-v0.1.39`

阶段目标：

- 为已有 Team 链路增加 request / response 协议
- 建立最小审批、认领与轮询机制
- 让 teammate 从“可通信”进入“可受控协作”

进入条件：

- Team 基础设施已经可运行
- lead 与 teammate 的消息链路已经稳定存在

退出条件：

- teammate 能按协议接收请求并返回响应
- 至少一种审批动作可被执行和记录
- 至少一种 auto claim 或 idle polling 流程跑通

推荐版本主题示例：

- `v0.1.36`：定义并落地 request / response 基础协议
- `v0.1.37`：补齐审批入口与状态变更记录
- `v0.1.38-v0.1.39`：补齐 auto claim、idle polling 或 identity reinjection 的最小闭环

不推荐的版本主题：

- 协议字段先行，但没有实际参与协议的 teammate
- 自治能力一口气做全，导致无法验证问题边界
- 在审批能力未落地前就扩展复杂策略系统

## 7. Stage D：Worktree Isolation

推荐版本区间：

- `v0.1.40+`

阶段目标：

- 建立 worktree registry
- 建立 task-worktree binding 生命周期
- 支持隔离目录执行与回收
- 补齐相关事件日志与审计链路

进入条件：

- Team 与 Protocol 已具备稳定承载对象
- 项目任务和执行边界已经足够清楚

退出条件：

- 任务可稳定绑定独立 worktree
- 指定 worktree 下的命令与文件操作可被正确执行
- worktree 生命周期可被查询、回收和审计

推荐版本主题示例：

- `v0.1.40`：worktree registry 与基础目录分配
- `v0.1.41`：task-worktree 绑定与状态联动
- `v0.1.42+`：隔离执行、回收与事件日志补齐

不推荐的版本主题：

- 只在任务字段里继续堆 `worktree` 文案，不做真实隔离
- 在 Team / Protocol 未稳定前直接做复杂隔离调度
- 一次性把 registry、调度、清理、审计全塞进同一版

## 8. 版本选题示例顺序

如果后续要继续按小版本推进，推荐的顺序可以直接参考：

1. `v0.1.31`：修正一个最影响首次上手的 CLI / README / check 摩擦点
2. `v0.1.32`：完成剩余高频误解的稳定化收口
3. `v0.1.33`：建立 `.team/` 与 teammate registry
4. `v0.1.34`：建立 inbox / message store
5. `v0.1.35`：打通 lead -> teammate 最小通信
6. `v0.1.36+`：围绕协议、审批、认领逐步推进
7. `v0.1.40+`：进入 worktree 隔离执行

## 9. 每次写新版本计划时的自检问题

在开始下一个版本前，先回答这五个问题：

- 这一版解决的唯一核心问题是什么
- 它和上一版的承接关系是什么
- 它对当前主线的直接贡献是什么
- 它的验收标准是否能被检查
- 它是否会把版本主题重新带回机械微调

如果这五个问题里有两项答不清楚，说明版本主题还不够好，不应直接进入开发。

## 10. 一句话执行口径

`v0.1.31+` 之后的每个版本，要么降低 `v0.1.30` 初代可用版的真实使用摩擦，要么推进 Team / Protocol / Worktree 这条后半段主线，不再默认沿着零碎字段继续递增。
