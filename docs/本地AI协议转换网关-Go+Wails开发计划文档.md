# 本地 AI 协议转换网关 Go + Wails 开发计划文档

## 1. 文档目标

本文档用于指导 `icoo_proxy` 项目的开发落地。项目采用 `Go + Wails v2` 技术方案，其中：

- `Go` 负责本地 AI 协议网关、配置加载、模型路由、HTTP 转发和后续协议转换核心逻辑
- `Wails v2` 负责桌面应用壳、可视化配置入口、运行状态展示和后续运维管理界面

本文档重点定义开发阶段、模块规划、当前首版落地范围与验收标准，避免项目一开始就把“桌面控制台”和“协议转换内核”做成两套割裂系统。

## 2. 总体开发策略

采用“先打通最小桌面可用闭环，再逐步增强协议转换能力”的迭代策略：

- 先让桌面应用能启动，并在应用内拉起本地 HTTP 代理
- 先支持三套协议入口和同协议透传
- 再补跨协议转换
- 先做本地单机可用，再做复杂治理能力
- 每个阶段都要求可运行、可验证、可演示

## 3. 技术方案

## 3.1 技术栈

- 桌面壳：Wails v2
- 后端语言：Go
- 前端：Wails 默认前端基础上定制的 Vanilla + CSS 界面
- 网关接口：本地 HTTP Server
- 配置来源：`.env` + 环境变量

## 3.2 架构原则

- 桌面壳和网关内核同进程协作，但模块边界清晰
- 协议适配和界面渲染分离
- 配置加载、路由决策、转发执行、状态展示各自独立
- 首版优先保证稳定可观察，不追求一次性支持全部协议细节

## 3.3 目录规划

建议按下面结构持续演进：

```text
icoo_proxy/
  main.go
  app.go
  wails.json
  .env.example
  cmd/
  frontend/
    src/
  internal/
    api/
    catalog/
    config/
    proxy/
    server/
```

## 4. 开发阶段规划

## Phase 1：桌面壳与本地代理骨架

### 4.1 目标

建立可启动的 Wails v2 桌面应用，并在应用启动时自动拉起本地 AI 网关。

### 4.2 范围

- Wails v2 项目骨架
- Go 配置加载
- 本地 HTTP 服务启动与关闭
- 桌面状态页
- 健康检查与基础管理接口

### 4.3 交付物

- 可运行的 `icoo_proxy` 桌面应用
- 可展示当前监听地址、路由和上游配置的首页
- 本地 `/healthz`、`/readyz`、`/admin/models`、`/admin/routes` 接口

### 4.4 验收标准

- `wails dev` 可以正常启动
- 应用启动后会自动拉起本地代理
- 桌面页能看到当前状态和配置摘要
- 应用关闭时本地代理能同步关闭

## Phase 2：三协议入口与同协议转发

### 4.5 目标

打通 Anthropic、OpenAI Chat Completions、OpenAI Responses 三套协议入口的同协议转发闭环。

### 4.6 范围

- `POST /v1/messages`
- `POST /v1/chat/completions`
- `POST /v1/responses`
- 对应前缀兼容路径
- 模型别名
- 请求 ID
- 基础鉴权
- 同协议上游转发

### 4.7 交付物

- 三套协议入口
- 统一模型路由目录
- 非流式与基础流式透传
- 请求日志和错误日志

### 4.8 验收标准

- Anthropic 请求可被转发到上游 Anthropic
- Chat 请求可被转发到上游 Chat
- Responses 请求可被转发到上游 Responses
- 模型别名能映射到真实模型名
- 客户端收到的错误结构保持协议风格

## Phase 3：跨协议转换引擎

### 4.9 目标

在统一语义模型基础上，支持有限的跨协议请求和响应转换。

### 4.10 范围

- Anthropic <-> OpenAI Responses
- Chat -> Anthropic
- 工具调用结构映射
- 停止原因和 usage 归一化

### 4.11 验收标准

- 至少一条跨协议链路可稳定调用
- 工具调用和基本 usage 信息可保留
- 无法映射的字段会明确报错

## Phase 4：桌面治理能力增强

### 4.12 目标

把桌面应用从“状态页”升级成“可操作控制台”。

### 4.13 范围

- 配置重载
- 上游联通性检测
- 路由编辑入口
- 最近请求摘要
- 错误状态与恢复提示

### 4.14 验收标准

- 用户可在桌面端查看当前配置摘要
- 可触发配置重载并看到结果
- 可查看最近错误和运行状态

## Phase 5：治理与可靠性增强

### 4.15 目标

增强本地网关的稳定性、安全性和可观测性。

### 4.16 范围

- 限流
- 并发控制
- 熔断与回退
- 调试模式
- 更多审计视图

## 5. 当前首版开发任务

当前这轮实际落地建议聚焦这些任务：

1. 生成 `icoo_proxy` 的 Wails v2 项目骨架。
2. 移除模板页，替换为代理状态控制台。
3. 实现 `.env` 配置加载。
4. 实现模型别名与默认路由解析。
5. 实现三套协议入口与同协议上游转发。
6. 实现桌面侧 `GetOverview` 与 `ReloadProxy` 方法。
7. 补充最小单元测试和构建验证。

## 6. 本版不做

- 完整跨协议转换
- 图片、音频等非文本接口
- 多租户权限体系
- 成本报表和复杂审计查询
- 远程控制台

## 7. 当前版本验收口径

当前首版可以判定为“已开始可用开发”的标准如下：

- `icoo_proxy` 已按 Wails v2 结构创建
- 应用可显示本地代理运行概览
- Go 后端可拉起 HTTP 网关
- 已暴露三套兼容入口
- 至少支持同协议透传
- 已提供模型别名配置和查看入口
- `go test ./...` 通过

## 8. 下一步建议

在首版骨架稳定后，下一步应优先继续做：

- Anthropic 与 Responses 的统一语义层
- Chat 与 Responses 的兼容转换
- 流式事件映射
- 工具调用转换
- 桌面端请求日志视图

这样可以保证 `icoo_proxy` 不是“只有桌面壳”的演示程序，也不是“只有后端代理”的黑盒服务，而是一个既能运行、又能被管理、还能逐步扩展成完整本地 AI 网关产品的基础盘。
