# v0.1.27 开发计划

## 1. 版本主题

为 `v0.1.30` 的可用初代版本交付补上一条明确的环境自检入口，先解决“用户拿到仓库后能不能快速判断自己是否能跑起来”这个最前置问题。

## 2. 为什么现在做

`v0.1.26` 之前的连续迭代主要在补 `task_audit` 的失败复盘可读性，但如果 `v0.1.30` 需要交付一个可以使用的初代版本，接下来的版本重点就不能继续停留在字段级微调上。第一步应先补齐启动前自检，让用户能在真正使用 Agent 前快速确认配置、工作区和基础运行条件。

## 3. 计划变更

- 为 CLI 增加一个面向初代可用版的环境自检入口
- 明确检查关键配置、工作区路径和基础运行前提
- README 补上自检命令与可用版上手顺序
- 为 `v0.1.30` 的交付目标建立倒排节奏

## 4. 本版不做

- 安装器或打包分发
- 完整的交互式初始化向导
- 发布级 UI 或桌面封装

## 5. 验收标准

- [x] CLI 新增环境自检入口
- [x] README 已补充自检命令与上手说明
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.27` 可以判定为已完成，CLI 现在已经提供 `assistant check` 环境自检入口，用户在真正运行 Agent 之前，可以先确认当前工作区、`.env`、client 模式、skills 目录以及关键运行目录是否处于可用状态。

完成依据：

- CLI 已新增 `assistant check` / `assistant doctor` 自检入口
- 自检输出会直接展示工作区、`.env`、client 模式、skills 目录和关键运行目录状态
- README 已补充自检命令与上手顺序
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前自检只负责判断最小运行前提，不自动修复复杂配置问题
- 当前自检不会主动探测外部网络或真实模型可用性
- 当前仍然没有完整初始化向导，下一步应继续补最小 happy path

## 7. 风险与停手线

- 如果自检入口一次性承载过多修复动作，会把本版从“可判断是否可用”拉成“半个安装器”
- 如果还继续沿着 `task_audit history` 字段链路微调，会稀释 `v0.1.30` 的交付节奏
- 因此本版应控制在“先把可用门槛显式化”而不是“顺手扩成完整初始化系统”
