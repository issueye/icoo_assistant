# v0.1.33 开发计划

## 1. 版本主题

正式切入 Team 主线，先落 `.team/` 基础结构、team config 和 teammate registry 的最小可运行骨架。

## 2. 为什么现在做

`v0.1.31-v0.1.32` 已经完成了 post-release hardening：首次上手路径、命令形式和 fake / real 模式边界都已经比较清楚。如果继续停留在 CLI 文案层优化，就会重新退回低价值的小修小补。

根据路线图，`v0.1.33` 应该开始进入 Stage B，也就是 Multi-Agent 基础设施的第一版。但当前还不适合直接做 inbox / message bus，因为在那之前，`.team/` 的持久化结构、lead 配置和 teammate roster 还没有落地。

因此，本版只回答一个问题：如何让 Team 主线第一次真正拥有可保存、可查询、可验证的基础状态。

## 3. 计划变更

- 新增 `internal/team` 模块，负责 `.team/config.json` 和 `.team/teammates/*.json`
- 自动初始化 `.team/` 基础目录与默认 `lead_id=lead` 配置
- 新增 `team_registry` 工具，支持 `get_config / update_config / create / get / list / update`
- 将 `.team/` 基础结构纳入 `assistant check`
- 更新 README 与工具边界说明，明确当前已经进入 Team 基础设施阶段

## 4. 本版不做

- teammate inbox / message bus
- lead -> teammate 实际消息通信
- 协议审批、自治认领与 worktree 隔离

## 5. 验收标准

- [x] `.team/`、`.team/config.json` 和 `.team/teammates/` 会自动初始化
- [x] `team_registry` 工具可查询/更新 team config，并管理 teammate roster
- [x] `assistant check` 能展示 team 基础结构是否 ready
- [x] README 已补充 Team 基础设施说明与边界
- [x] `go test ./...` 通过

## 6. 完成度评估

`v0.1.33` 可以判定为已完成。本版是 Team 主线的起点版，重点不是“让多个 Agent 立刻通信”，而是先把后续多 Agent 能依赖的持久化底座建出来。

本版完成情况如下：

- 新增 `internal/team` 模块，持久化 `.team/config.json` 和 `.team/teammates/teammate_<id>.json`
- 默认会初始化 `lead_id=lead` 的 team config
- 新增 `team_registry` 工具用于配置 `lead` 和管理 teammate roster
- `tool_catalog` 已能解释 `team_registry` 的职责边界
- `assistant check` 已纳入 `.team/`、teammate registry 目录和 team config 摘要
- README 已新增 Team 基础设施说明、边界和示例命令

当前边界判断：

- 这版已经满足 Stage B 第一小步“team config + teammate registry”
- 还没有进入消息总线，因此“队友已存在”不等于“队友已可通信”
- 下一版应围绕 inbox / message store 展开，而不是继续在 roster 字段上机械追加

## 7. 风险与停手线

- 如果继续在 `team_registry` 上补零散字段，会很快重演之前的小版本漂移问题
- 如果现在直接跳到复杂协议层，会缺少稳定的消息承载对象
- 因此本版停在“Team 状态可持久化、可查询、可解释”，把下一步空间留给 `v0.1.34` 的 inbox / message store
