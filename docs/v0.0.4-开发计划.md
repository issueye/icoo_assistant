# v0.0.4 开发计划

## 1. 版本定位

`v0.0.4` 聚焦把已经存在的项目级任务能力从“内部模块”推进到“正式可操作入口”。核心目标是为任务系统补齐工具层接入和基础展示能力，让 Agent 可以直接创建、查询和更新项目任务。

## 2. 版本目标

本版本计划交付以下能力：

- 项目任务管理工具入口
- 任务创建、查询、列表、状态更新的工具调用能力
- 与现有 `todo`、`background` 的职责边界说明
- 相关测试和文档

## 3. 范围定义

### 3.1 纳入 `v0.0.4`

- 新增项目任务工具
- 将 `internal/task` 接入 `app`
- 提供任务列表和单任务读取输出格式
- 提供状态更新入口
- 文档与测试

### 3.2 暂不纳入 `v0.0.4`

- 自动任务拆解
- 任务自动认领
- 多 Agent 团队协作
- 协议审批
- worktree 绑定

## 4. 设计重点

### 4.1 命名与职责

- 会话内短期规划继续由 `todo` 负责
- 项目级持久化任务由单独任务工具负责
- 后台执行继续由 `background` 负责

### 4.2 工具能力

建议支持以下动作：

- `create`
- `get`
- `list`
- `update`
- `update_status`

## 5. 任务拆解

### A. 工具接入

- 创建任务工具定义
- 接入 `app` registry
- 统一输出格式

### B. 测试验证

- 创建任务工具测试
- 状态更新测试
- 与现有工具并存的集成验证

### C. 文档收口

- 更新 README
- 更新版本文档
- 明确 `todo` / `task` / `background` 分工

## 6. 验收清单

- [ ] Agent 可通过工具创建项目任务
- [ ] Agent 可通过工具查询和列出项目任务
- [ ] Agent 可通过工具更新任务状态
- [ ] 测试通过
- [ ] 文档更新

## 7. 风险提示

- `task` 名称当前已被子代理委托占用，工具命名必须避免冲突
- 工具输出如果过于冗长，会影响主循环上下文
- 项目任务与后台任务的关系如果过早耦合，后续调度层会难以演进
