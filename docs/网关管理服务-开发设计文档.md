# 网关管理服务开发设计文档

## 1. 文档目标

本文档基于《网关管理服务需求分析文档》，给出 `icoo_assistant` 网关管理服务的开发设计方案，重点说明系统职责划分、模块结构、数据模型、接口设计、分阶段落地方式与验收策略，确保后续开发能够稳定推进，不再把 Team / Conversation / Skill 治理逻辑重新塞回本地 CLI。

## 2. 设计原则

服务设计遵循以下原则：

- 控制面与执行面分离
- 资源模型统一
- 单对话与 Team 对话统一建模
- 配置对象与运行对象分离
- 先交付可用的管理闭环，再做复杂自治和协议
- 先保证 API 和数据模型稳定，再扩展 UI 和高级调度

## 3. 系统定位

建议把系统划分为两层：

### 3.1 执行面

由现有本地 Runtime 承担，主要负责：

- 模型调用
- 工具调用
- 工作区内文件和命令执行
- 本地任务与审计

### 3.2 控制面

由新的网关管理服务承担，主要负责：

- Skill 管理
- Agent 模板和实例管理
- Team 管理
- 单对话与 Team 对话管理
- 对话路由与编排
- 统一审计和资源查询

## 4. 总体架构

建议采用“API 网关 + 领域服务 + 持久层”的单体优先架构，先做可演进模块化单体，后续再按压力拆分。

```text
clients
  |- assistant CLI
  |- future web console
  |- external services

gateway management service
  |- api layer
  |- auth layer
  |- skill module
  |- agent module
  |- team module
  |- conversation module
  |- routing/orchestration module
  |- audit module
  |- persistence layer

execution runtimes
  |- local assistant runtime
  |- future remote agent runtimes
```

## 5. 建议模块划分

建议在新的服务模块中拆分以下目录：

```text
gateway/
  cmd/
    gateway/
      main.go
  internal/
    api/
    auth/
    skill/
    agentprofile/
    agentinstance/
    team/
    conversation/
    routing/
    audit/
    storage/
    config/
```

模块职责建议如下：

- `api`：HTTP 路由、请求校验、响应格式
- `auth`：鉴权、中间件、调用方识别
- `skill`：Skill 元数据与版本管理
- `agentprofile`：Agent 模板管理
- `agentinstance`：Agent 实例、状态、心跳
- `team`：Team 与成员关系管理
- `conversation`：单对话和 Team 对话统一管理
- `routing`：消息路由、入口 Agent 选择、内部消息分发
- `audit`：审计事件入库与查询
- `storage`：数据库访问抽象
- `config`：服务配置加载

## 7. 对话与路由模型

## 7.1 统一对话模型

单对话和 Team 对话建议共用一张 `conversation` 主表，只通过 `mode` 区分。

这样做的好处：

- API 一致
- 历史查询一致
- 审计模型一致
- 未来做跨模式迁移更容易

## 7.2 外部消息与内部消息分层

建议把消息分为三类：

- `external`：用户与系统看到的主消息流
- `internal`：Team 内部 Agent 间消息
- `system`：系统路由、状态变化、失败说明

这样可以避免 Team 对话把所有内部协同细节直接暴露给用户。

## 7.3 Team 对话基础路由策略

第一阶段建议只支持最简单的路由策略：

1. 外部消息进入 Team 对话
2. 路由到 `entry_agent`
3. `entry_agent` 可选择直接回复，或分派给其他成员
4. 其他成员返回内部消息
5. 由入口 Agent 或汇总 Agent 给出最终外部回复

复杂审批、自治和自动认领不放在第一阶段。

## 8. API 设计建议

建议先提供 REST API，后续如果需要实时推送，再补 SSE 或 WebSocket。

## 8.1 Skill API

- `POST /api/v1/skills`
- `GET /api/v1/skills`
- `GET /api/v1/skills/{id}`
- `PATCH /api/v1/skills/{id}`
- `POST /api/v1/skills/{id}/activate`
- `POST /api/v1/skills/{id}/deactivate`

## 8.2 Agent API

- `POST /api/v1/agent-profiles`
- `GET /api/v1/agent-profiles`
- `GET /api/v1/agent-profiles/{id}`
- `PATCH /api/v1/agent-profiles/{id}`
- `POST /api/v1/agent-instances`
- `GET /api/v1/agent-instances`
- `GET /api/v1/agent-instances/{id}`
- `POST /api/v1/agent-instances/{id}/heartbeat`
- `POST /api/v1/agent-instances/{id}/disable`

## 8.3 Team API

- `POST /api/v1/teams`
- `GET /api/v1/teams`
- `GET /api/v1/teams/{id}`
- `PATCH /api/v1/teams/{id}`
- `POST /api/v1/teams/{id}/members`
- `PATCH /api/v1/teams/{id}/members/{memberId}`
- `DELETE /api/v1/teams/{id}/members/{memberId}`

## 8.4 Conversation API

- `POST /api/v1/conversations`
- `GET /api/v1/conversations`
- `GET /api/v1/conversations/{id}`
- `GET /api/v1/conversations/{id}/messages`
- `POST /api/v1/conversations/{id}/messages`
- `GET /api/v1/conversations/{id}/runs`

建议 `POST /api/v1/conversations` 通过 `mode` 区分：

- `mode=single` 时必须提供 `target_agent_id`
- `mode=team` 时必须提供 `target_team_id`

## 8.5 Audit API

- `GET /api/v1/audit-events`
- `GET /api/v1/audit-events/{id}`

## 9. 存储方案建议

第一阶段建议使用：

- `PostgreSQL`：主业务数据
- `Redis`：可选，用于会话短状态、路由队列、幂等键

如果要尽快起步，也可以先用单数据库方案：

- 全量先落 PostgreSQL
- 队列与心跳状态先走数据库表

不建议第一阶段继续沿用 `.team/` 这类本地目录作为 Team 管理核心存储。

## 10. 与本地 Runtime 的集成方式

建议采用“网关下发上下文，Runtime 执行并回报结果”的模式。

### 10.1 下发内容

网关可向 Runtime 下发：

- 对话上下文
- 目标 Agent 模板配置
- 可用 Skill 列表
- Team 上下文或路由指令

### 10.2 回传内容

Runtime 回传：

- 最终回复
- 内部路由请求
- 执行状态
- 失败信息
- 审计摘要

### 10.3 集成边界

本地 Runtime 不负责：

- Skill 元数据治理
- Agent / Team 持久化管理
- Team 对话总控

本地 Runtime 负责：

- 按网关下发的上下文执行当前轮任务
- 回传执行结果

## 11. 分阶段开发计划

## Phase 1：资源管理底座

目标：

- 建立服务骨架
- 落地 Skill / Agent / Team 基础 CRUD
- 统一审计模型

交付：

- 服务启动入口
- 基础配置
- 数据库迁移
- Skill / Agent / Team API
- 基础审计查询

验收：

- 可以创建、修改、查询 Skill / Agent / Team
- 审计事件正常落库

## Phase 2：单对话管理

目标：

- 建立单对话模型
- 打通会话创建、续聊、历史查询

交付：

- Conversation 与 Message 表
- 单对话 API
- 基础 Run 记录

验收：

- 能创建单对话
- 能持续追加消息
- 能查询消息与运行历史

## Phase 3：Team 对话基础路由

目标：

- 建立 Team 对话模式
- 支持入口 Agent 路由与基础内部消息流

交付：

- Team 对话 API
- 内外消息分层
- 基础路由模块

验收：

- 能创建 Team 对话
- 能记录内部路由消息
- 能返回最终对外回复

## Phase 4：Runtime 集成

目标：

- 打通网关服务与本地 Runtime 的交互
- 支持 Agent 实例心跳和执行结果回传

交付：

- Agent 心跳接口
- Runtime 执行回传接口
- 运行状态聚合视图

验收：

- 能看到 Agent 在线状态
- 能把一次执行结果回收到对话和审计系统

## Phase 5：治理增强

目标：

- 提升稳定性与管理能力
- 为未来 Web 管理端铺路

交付：

- 统计视图
- 资源启停
- 常见失败补偿
- 基础权限控制

验收：

- 关键资源可以停用
- 常见错误可定位
- Web 管理端接入无需重做核心 API

## 12. 测试策略

测试建议分四层：

- 单元测试：领域对象、状态流转、路由规则
- 集成测试：API + 数据库
- 契约测试：网关与 Runtime 之间的请求响应结构
- 冒烟测试：Skill、Agent、Team、单对话、Team 对话的最小闭环

第一阶段必须覆盖：

- 资源 CRUD
- 会话创建与续聊
- Team 成员绑定
- Team 对话入口路由
- 审计事件记录

## 13. 风险与应对

### 风险一：控制面和执行面边界再次混乱

应对：

- 开发前先固定模块边界
- Team 相关治理逻辑只允许进入网关服务

### 风险二：单对话和 Team 对话形成两套模型

应对：

- 强制复用 `conversation` 主模型
- 只在 `mode` 和路由层做差异

### 风险三：过早追求复杂自治导致架构空转

应对：

- 第一阶段只做基础路由
- 审批、自治、自动认领后置

### 风险四：继续依赖本地文件目录做平台级存储

应对：

- 尽早建立数据库模型
- `.team/` 只允许作为历史遗留运行目录，不再作为未来主存储

## 14. 推荐的近期落地顺序

建议下一步按这个顺序开始开发：

1. 先搭建 `gateway` 服务骨架和配置体系
2. 先完成 Skill / Agent / Team 的数据库模型和 CRUD API
3. 再完成统一 Conversation 模型
4. 再补 Team 对话基础路由
5. 最后对接本地 Runtime

这个顺序的意义是先把管理面站稳，再去接执行面，避免又回到“为了跑通一个样例，把状态写死在本地进程里”的旧路径。

## 15. 一句话结论

网关管理服务的正确开发方式，不是先补一堆 Team 小功能，而是先建立统一资源模型与会话模型，再用清晰的控制面架构逐步接住 Skill、Agent、Team 和 Conversation 的治理职责。
