# v0.1.17 开发计划

## 1. 版本主题

让 `task_audit summary` 在指出最近变化点后，进一步给出更直接的最新样本对比入口提示。

## 2. 为什么现在做

`v0.1.16` 已经能告诉用户“最近主要变了什么”，下一步最自然的问题就是“应该优先对比哪两条样本”。这一版继续停留在摘要导航层，不直接扩展成更重的诊断结论或自动分析。

## 3. 计划变更

- 为优先失败原因增加更直接的双样本对比提示
- 让最新样本与上一条样本的查看路径更明确
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 自动生成根因分析
- 跨任务对比入口
- 面向图表的可视化摘要

## 5. 验收标准

- [x] `task_audit summary` 新增一个面向双样本对比的提示字段
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.17` 可以判定为已完成，当前任务侧审计已经能在指出最近变化点后，直接给出最新样本与上一条样本的查看入口，帮助用户更快进入下一步历史对比。

完成依据：

- 已在 `task_audit summary` 中新增 `priority_failure_compare_target`
- 当前输出会基于最近同类失败样本给出 `limit=1` 或 `limit=2` 的建议历史查询入口
- README 与 `tool_catalog` 已同步补上双样本对比入口说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前双样本对比入口仍然只覆盖优先失败原因
- 当前入口只负责告诉用户“先看哪两条”，不会直接重排 history 输出
- 当前对比顺序仍然依赖用户进入 `history` 后自行读取条目内容

## 7. 风险与停手线

- 如果对比入口提示同时塞入过多字段，summary 会再次变重
- 如果入口提示开始代替历史查询，会让摘要和 history 的职责边界变模糊
- 因此本版应控制在“告诉用户先看哪两条”而不是“替用户完成详细对比”
