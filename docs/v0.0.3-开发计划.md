# v0.0.3 开发计划

## 1. 版本定位

`v0.0.3` 聚焦把当前已经具备的单 Agent 能力和任务持久化能力向“可持续执行”推进。核心目标是补齐后台任务执行和结果回流，让系统开始支持更真实的长耗时开发任务。

## 2. 版本目标

本版本计划交付以下能力：

- 后台命令执行器
- 后台任务状态跟踪
- 主循环中的后台结果通知
- 与现有任务模块的基础协同约定
- 相关测试和文档

## 3. 范围定义

### 3.1 纳入 `v0.0.3`

- 新增 `internal/background` 模块
- 后台命令启动与结果存储
- 轮询或回流机制
- 后台任务基础状态模型
- 文档与测试

### 3.2 暂不纳入 `v0.0.3`

- 多 Agent 团队协作
- 审批协议
- 自动认领任务
- worktree 隔离
- Web 控制台

## 4. 设计重点

### 4.1 核心原则

- 后台任务不能阻塞主 Agent 循环
- 后台任务结果必须能回流到主上下文
- 首版优先保证可靠和可测，不急于做复杂调度器

### 4.2 与现有模块的衔接

- 复用现有命令执行逻辑的安全约束
- 为后续任务系统预留 `owner`、`taskID` 或关联字段
- 保持 `todo`、`task`、`background` 职责边界清晰

## 5. 任务拆解

### A. 模块骨架

- 创建 `internal/background`
- 定义后台任务结构体和状态
- 定义管理器接口

### B. 核心能力

- 启动后台命令
- 查询后台任务状态
- 记录标准输出、错误和退出结果
- 提供结果通知或轮询接口

### C. 接入与文档

- 在 Agent 主循环预留后台结果注入点
- 补充测试
- 更新 README 和版本文档

## 6. 验收清单

- [x] 可以启动后台命令且不阻塞主流程
- [x] 可以查询后台任务状态
- [x] 后台任务完成后可获取输出结果
- [x] 测试通过
- [x] 文档更新

## 7. 完成度评估

### 7.1 当前结论

`v0.0.3` 可以判定为已完成，后台任务模块、工具入口和主循环结果回流都已经具备最小可用闭环。

完成依据：

- 已新增 `internal/background` 模块
- 已支持后台任务启动、查询、列表与完成通知
- 已新增 `background` 工具并接入 `app`
- Agent 主循环已支持轮询后台完成结果并注入消息
- `go test ./...` 已通过
- README 已同步更新后台任务说明

### 7.2 当前边界

- 后台能力目前通过工具层使用，尚未提供独立 CLI 子命令
- 后台任务结果采用本地文件存储，暂未处理跨进程争抢或更复杂调度
- 项目级任务与后台任务之间还只是基础字段约定，尚未形成统一任务编排
## 8. 风险提示

- 后台任务一旦与主循环耦合过深，会让后续多 Agent 扩展变复杂
- Windows 和 Unix 的命令执行差异需要沿用现有跨平台策略
- 结果回流机制如果设计过早过重，后续协议层会很难调整
