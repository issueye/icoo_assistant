# 基于 learn-claude-code 教程的 Go 版 Agent 系统需求分析文档

## 1. 文档目的

本文档基于 `learn-claude-code/docs/zh` 下 12 篇教程内容，提炼一个可使用 Golang 实现的 Agent Harness/Assistant 系统需求，输出面向开发落地的需求分析结果，为后续架构设计、任务拆解和迭代开发提供依据。

## 2. 来源材料分析结论

教程内容从 `s01` 到 `s12` 逐步构建了一个完整的 Agent 执行框架，核心演进路径如下：

1. **Agent Loop**：建立模型调用、工具执行、结果回注的基本闭环。
2. **Tool Use**：引入专用工具和分发机制，降低全依赖 Shell 的风险。
3. **TodoWrite**：引入过程规划能力，避免多步骤任务偏航。
4. **Subagent**：通过上下文隔离处理复杂任务，降低主会话污染。
5. **Skill Loading**：按需加载领域知识，控制系统提示长度。
6. **Context Compact**：提供上下文压缩与转储机制，支持长会话。
7. **Task System**：将任务从会话内清单升级为可持久化、可依赖管理的任务图。
8. **Background Tasks**：支持耗时命令异步执行与通知回流。
9. **Agent Teams**：支持多 Agent 持久协作。
10. **Team Protocols**：通过请求-响应协议实现审批、关机等协作约束。
11. **Autonomous Agents**：支持 Agent 自主认领任务和空闲轮询。
12. **Worktree Isolation**：通过任务与 worktree 绑定实现隔离式并行开发。

**结论**：该教程本质上描述的是一个“面向编码任务的 Agent 运行时系统”，不是单一聊天机器人。Go 实现应围绕“调度、工具、状态、持久化、并发、隔离”六个核心能力建设。

## 3. 产品目标

开发一个基于 Golang 的本地/服务端 Agent 执行系统，能够：

- 与大模型进行多轮交互。
- 安全调用本地工具与文件系统能力。
- 管理复杂开发任务的计划、执行和状态追踪。
- 支持子 Agent、团队 Agent、后台任务和隔离工作目录。
- 在长时运行场景下保持可恢复、可审计、可扩展。

## 4. 目标用户

### 4.1 主要用户
- 使用 AI 辅助开发的后端/全栈工程师
- 需要构建自定义编码代理平台的技术团队
- 需要实验多 Agent 协作机制的研发人员

### 4.2 使用场景
- 自动化代码生成与重构
- 项目级文件读取、编辑、测试执行
- 多步骤开发任务规划与追踪
- 并行执行测试、构建、检查等耗时任务
- 多 Agent 协作完成复杂开发目标
- 在不同 worktree 中并行处理多个需求/缺陷

## 6. 核心业务流程

### 6.1 单 Agent 执行主流程
1. 用户输入任务。
2. 系统生成消息列表并调用模型。
3. 模型返回普通响应或工具调用。
4. 系统执行工具并回注结果。
5. 循环直到模型不再请求工具。
6. 根据上下文大小决定是否压缩和落盘。

### 6.2 多任务执行流程
1. 用户创建目标任务。
2. 系统拆解为多个 Task，并设置依赖。
3. Agent 或 Teammate 认领任务。
4. 任务进入执行中状态。
5. 如需隔离开发环境，则创建 worktree 并绑定任务。
6. 任务完成后更新状态，自动解锁后续任务。

### 6.3 团队协作流程
1. 领导 Agent 创建/拉起队友 Agent。
2. 通过 inbox 发送消息或请求。
3. 队友处理任务、发送结果或审批响应。
4. 空闲 Agent 轮询新消息和未认领任务。
5. 超时无工作则进入关闭。

## 7. 功能需求

## 7.1 P0：基础闭环能力

### FR-001 Agent 对话循环
系统必须支持基于消息列表的循环式模型调用机制，直到 `stop_reason != tool_use` 或达到安全轮数上限。

### FR-002 工具调用分发
系统必须提供统一工具注册与分发机制，按工具名映射到处理函数，不允许在主循环中写死工具分支。

### FR-003 本地命令执行
系统必须支持运行受控 Shell 命令，并限制工作目录与超时时间。

### FR-004 文件读取/写入/编辑
系统必须支持工作区内文件的读取、写入与文本替换，禁止路径逃逸。

### FR-005 安全路径校验
所有文件工具必须经过路径归一化与工作区边界校验。

## 7.2 P0：过程管理能力

### FR-006 Todo 管理
系统必须支持会话级 Todo 列表，至少包含 `pending / in_progress / completed` 三种状态，且同一时刻最多一个任务处于 `in_progress`。

### FR-007 Todo 提醒机制
当 Agent 在若干轮对话中未更新 Todo 时，系统应支持注入提醒信息。

### FR-008 上下文压缩
系统必须支持旧工具结果轻量压缩，以及超阈值时的摘要压缩。

### FR-009 会话转储
系统必须将压缩前的完整对话写入磁盘，用于恢复与审计。

## 7.3 P1：任务持久化能力

### FR-010 Task 持久化
系统必须支持将任务持久化到磁盘，任务至少包含：ID、主题、状态、blockedBy、owner、worktree。

### FR-011 任务依赖
系统必须支持任务间依赖管理；当前置任务完成后，自动解除其他任务中的阻塞依赖。

### FR-012 任务查询
系统必须支持任务创建、读取、更新、列表查询等能力。

## 7.4 P1：复杂任务处理能力

### FR-013 Subagent 派发
系统必须支持启动上下文隔离的子 Agent 执行子任务，并仅向父 Agent 返回摘要结果。

### FR-014 Skill 按需加载
系统必须支持从本地 Skill 目录扫描技能定义，并按需将技能内容注入当前对话。

### FR-015 后台任务
系统必须支持将长耗时命令放入后台执行，并在后续轮次中通知执行结果。

## 7.5 P2：多 Agent 协作能力

### FR-016 Team 成员管理
系统应支持创建、查询、维护多个长期存活的队友 Agent，并记录其角色与状态。

### FR-017 消息总线
系统应支持基于文件或其他轻量持久层的 Agent 间收发消息机制。

### FR-018 协议化请求/响应
系统应支持带唯一请求 ID 的请求-响应机制，至少覆盖计划审批与优雅关闭。

### FR-019 自主认领任务
空闲 Agent 应支持扫描任务看板并自动认领可执行任务。

### FR-020 身份重注入
在会话压缩后，系统应支持重新注入 Agent 身份与角色信息。

## 7.6 P2：隔离执行能力

### FR-021 Worktree 生命周期管理
系统应支持创建、绑定、查询、保留、删除 worktree。

### FR-022 Task-Worktree 绑定
系统应支持任务与 worktree 的双向关联，创建绑定时自动将任务置为执行中。

### FR-023 隔离目录执行
系统应支持在指定 worktree 目录下执行命令和文件操作。

### FR-024 生命周期事件记录
系统应记录 worktree 与任务状态变化事件，支持后续恢复与排查。

## 8. 非功能需求

### NFR-001 并发能力
Go 实现需充分利用 goroutine + channel/mutex，支持后台任务、多 Agent、通知队列并发处理。

### NFR-002 可恢复性
核心状态必须持久化到磁盘；系统重启后可恢复任务、转储、worktree 索引等现场。

### NFR-003 安全性
- 文件访问必须限制在允许目录内。
- 命令执行必须可配置超时。
- 敏感操作需提供显式确认机制。
- 不允许隐式删除用户数据。

### NFR-004 可扩展性
工具、技能、协议、任务状态等必须可扩展，避免硬编码耦合。

### NFR-005 可观测性
应提供结构化日志，记录模型调用、工具调用、状态变更、错误、事件。

### NFR-006 性能
- 普通工具调用应具备毫秒到秒级响应。
- 后台任务不得阻塞主循环。
- 长会话下上下文大小应受控。

## 9. 建议的 Go 模块划分

1. `internal/agent`：主循环、消息编排、stop condition。
2. `internal/tools`：工具定义、注册、分发、执行器。
3. `internal/workspace`：路径校验、文件操作、目录边界控制。
4. `internal/todo`：会话级待办管理。
5. `internal/compact`：上下文压缩与 transcript 管理。
6. `internal/task`：任务图、依赖处理、持久化。
7. `internal/background`：后台命令执行与通知。
8. `internal/skill`：技能扫描、元数据解析、内容注入。
9. `internal/team`：队友生命周期、消息总线、协议。
10. `internal/worktree`：worktree 注册、绑定、移除、事件。
11. `internal/storage`：JSON/JSONL 文件读写抽象。
12. `cmd/assistant`：CLI 入口。

## 11. 技术选型建议（Go）

- 语言版本：Go 1.23+
- CLI：`cobra`（可选）
- 配置：`yaml` / `json`
- 并发：goroutine、channel、context、sync.Mutex
- 命令执行：`os/exec`
- 文件系统：`os`、`filepath`
- 结构化日志：`log/slog`
- 持久化：JSON / JSONL 文件优先，后续可升级 BoltDB/SQLite
- 模型接入：先抽象 LLM Client Interface，再对接具体模型 SDK

## 12. 风险与难点分析

### 12.1 上下文与状态一致性
问题：压缩后可能导致 Agent 忘记身份、任务阶段或最近结果。
应对：保留摘要、身份重注入、重要状态外置到磁盘。

### 12.2 并发写文件冲突
问题：多个 goroutine/Agent 同时写任务、收件箱、事件文件时可能冲突。
应对：增加文件级锁或进程内锁，统一写入入口。

### 12.3 Worktree 与任务状态不一致
问题：worktree 创建成功但任务未绑定，或任务完成但 worktree 未清理。
应对：使用事务式更新顺序与事件日志补偿。

### 12.4 工具执行安全边界
问题：Shell 和文件工具是高风险入口。
应对：工作区沙箱、显式确认、高风险工具白名单。

## 13. 验收标准

### P0 验收
- 可完成一次“读取文件 -> 修改文件 -> 再读取验证”的完整闭环。
- Todo 状态可被正确维护。
- 上下文超限后可自动压缩并保留转储文件。

### P1 验收
- 可创建多任务并建立依赖。
- 可启动 Subagent 完成子任务并返回摘要。
- 可运行后台任务且主循环继续工作。

### P2 验收
- 可创建多个队友 Agent 并互相收发消息。
- 可发起审批请求并收到带 request_id 的响应。
- 可自动认领任务并在独立 worktree 中执行。

## 15. 总结

教程内容非常适合转化为一个分阶段实现的 Go Agent Runtime。需求核心不是“聊天”，而是“围绕编码任务的可执行智能体平台”。因此，Go 版本建设重点应放在执行闭环、持久化状态、并发调度和隔离运行环境上，再逐步扩展到多 Agent 协作与自治能力。
