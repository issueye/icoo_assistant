# v0.1.23 开发计划

## 1. 版本主题

让 `task_audit history` 在聚焦失败查看时补上一条 latest failure signature 提示，减少用户继续在条目里手动寻找本次失败的错误签名。

## 2. 为什么现在做

`v0.1.22` 已经能让用户更快看到最新失败执行了什么命令，下一步最自然的问题就是“这条最新失败的错误模式是什么”。这一版继续停留在 history 可读性层，不扩展到新的推断逻辑。

## 3. 计划变更

- 为 `task_audit history` 增加一个轻量 latest failure signature 提示
- 让失败历史视图更容易快速对齐当前失败记录与错误模式
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 错误级 diff
- 自动修复建议
- 更复杂的失败模式分析

## 5. 验收标准

- [x] `task_audit history` 新增 latest failure signature 提示
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.23` 可以判定为已完成，当前任务侧审计已经能在 history 视图里通过 `latest_failure_signature` 直接提示最新失败样本的错误签名，让用户在定位到最新失败和执行命令后，也能立刻对齐当前错误模式。

完成依据：

- 已在 `task_audit history` 中新增 `latest_failure_signature`
- 当前 `status=failed` 与 `reason=<reason>` 的失败聚焦历史查看都会直接提示最新失败签名
- README 与 `tool_catalog` 已同步补上 latest failure signature 提示说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 `latest_failure_signature` 仍然只是轻量提示，不承担错误解释
- 当前只提示最新失败签名，不扩展到错误级 diff
- 当前仍然依赖 history 返回中的最近一条失败记录

## 7. 风险与停手线

- 如果 latest failure signature 提示与现有条目、pair 摘要重复过多，会削弱 history 的紧凑性
- 如果提示开始承载解释语义，会和 summary 的趋势/变化点字段混在一起
- 因此本版应控制在“更快看到错误模式”而不是“解释错误为什么发生”
