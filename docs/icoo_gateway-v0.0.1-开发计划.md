# icoo_gateway v0.0.1 开发计划

## 1. 版本定位

`icoo_gateway v0.0.1` 定位为网关管理服务的首个“可开发、可演示、可继续扩展”的控制面骨架版本。

这个版本的目标不是一次性做成完整产品，而是为 `v0.1.0` 可交付版建立稳定基础，重点解决以下问题：

- 先把资源模型和 API 骨架站稳
- 先把基础治理动作补齐
- 先把审计与运行扩展位预留出来
- 不在 `v0.0.1` 过早引入复杂 Team 自治和完整 Runtime 编排

## 2. 与 v0.1.0 的关系

`v0.1.0` 目标是形成可交付产品，因此至少需要具备：

- 稳定的资源治理 API
- 基本可追踪的运行与审计能力
- 基础可用的 Team 对话治理能力
- 清晰的控制面与执行面边界
- 适合后续 Web 管理端接入的稳定接口形态

因此，`v0.0.1` 的职责是“打基础”，不追求功能大而全，而要尽量减少后续返工。

## 3. v0.0.1 核心目标

本版本计划完成以下四件事：

- 建立稳定的网关服务骨架
- 落地核心资源的最小可用 CRUD 和基础治理动作
- 建立最小审计查询骨架
- 保持统一对话模型和 Team 路由最小闭环可继续扩展

## 4. 范围定义

### 4.1 纳入 v0.0.1

- `icoo_gateway` 独立服务入口
- 基础配置加载与 HTTP 服务
- `skill / agent_profile / agent_instance / team / conversation` 资源基础 API
- `team member` 基础关系管理
- `team conversation` 最小路由骨架
- `agent_instance heartbeat`
- 资源更新、启停类基础治理 API
- `audit-events` 最小查询能力
- 基础单元测试与说明文档

### 4.2 暂不纳入 v0.0.1

- PostgreSQL 持久化落地
- Redis 或队列能力
- 完整 `Run` 模型
- Runtime 执行回传闭环
- Team 真正的多成员协作编排
- 权限控制与鉴权中间件
- Web 控制台

## 5. 设计原则

- 先管理面，后执行面
- 先统一资源模型，后复杂流程编排
- 先把 API 形态稳定下来，再替换底层存储
- 先补治理动作，再扩展高级自治
- 为 `v0.1.0` 的可交付目标预留扩展位

## 6. 版本任务拆解

### A. 网关骨架稳定化

- 保持当前服务启动、配置加载、健康检查稳定
- 保持统一 API 路由入口清晰
- 维持 `single/team` 统一 conversation 模型

### B. 资源治理补齐

- 为 `skill` 补齐 `PATCH /activate / deactivate`
- 为 `agent_profile` 补齐 `PATCH`
- 为 `agent_instance` 补齐 `disable`
- 为 `team` 补齐 `PATCH`
- 为 `team member` 补齐 `PATCH / DELETE`

### C. 审计骨架

- 新增 `audit` 模块
- 新增 `GET /api/v1/audit-events`
- 新增 `GET /api/v1/audit-events/{id}`
- 对关键资源变更动作记录最小审计事件

### D. 测试与文档

- 为新增 API 补测试
- 保持 `go test ./...` 通过
- 更新 `README` 和版本文档

## 7. v0.0.1 验收标准

- 可启动 `icoo_gateway` 服务
- 可创建、查询、更新、启停 Skill / Agent / Team 基础资源
- 可新增、更新、删除 Team 成员
- 可查询基础审计事件
- Team 对话最小消息分层与路由行为仍然可用
- `go test ./...` 全绿

## 8. 不在 v0.0.1 解决的问题

以下问题必须明确留到后续版本，不在本版本硬塞进去：

- 数据持久化
- 真实 Runtime 回传
- Team 最终对外回复闭环
- 复杂审批、自治、自动认领
- Web 管理端

## 9. 通往 v0.1.0 的版本路径

建议按下面节奏推进：

- `v0.0.1`：资源治理骨架 + 审计骨架
- `v0.0.2`：存储抽象与持久化接入
- `v0.0.3`：Run 模型与运行查询
- `v0.0.4`：Runtime 单对话执行回传闭环
- `v0.0.5`：Team 对话最小外部回复闭环
- `v0.1.0`：可交付版，形成稳定治理、审计、基础联调与可演示路径

## 10. 一句话结论

`icoo_gateway v0.0.1` 的正确目标，不是做成最终产品，而是把“资源治理 API 形态、最小审计骨架、统一会话模型”三件事打稳，为 `v0.1.0` 可交付版减少返工。
