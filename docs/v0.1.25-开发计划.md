# v0.1.25 开发计划

## 1. 版本主题

让 `task_audit history` 在聚焦失败查看时补上一条 latest failure entry hint，减少用户继续在条目列表里人工确认最新失败对应的是哪一行。

## 2. 为什么现在做

`v0.1.24` 已经能让用户更快看到最新失败的时间点，下一步最自然的问题就是“这条最新失败在当前返回结果里具体对应哪一行”。这一版继续停留在 history 可读性层，不扩展到新的推断逻辑。

## 3. 计划变更

- 为 `task_audit history` 增加一个轻量 latest failure entry hint
- 让失败历史视图更容易快速把 latest 提示和具体条目对应起来
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 自动高亮 UI
- 更复杂的历史排序策略
- 新的诊断或修复建议

## 5. 验收标准

- [x] `task_audit history` 新增 latest failure entry hint
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.25` 可以判定为已完成，当前任务侧审计已经能在 history 视图里通过 `latest_failure_entry` 直接提示最新失败样本对应的是当前返回结果里的第几条，减少用户还要在条目列表里人工对位一次。

完成依据：

- 已在 `task_audit history` 中新增 `latest_failure_entry`
- 当前 `status=failed` 与 `reason=<reason>` 的失败聚焦历史查看都会直接提示最新失败对应的返回条目与 `job_id`
- README 与 `tool_catalog` 已同步补上 latest failure entry 提示说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 `latest_failure_entry` 仍然只是轻量定位提示，不承担失败解释
- 当前只提示最新失败对应的返回条目，不扩展到自动高亮或 UI 层标记
- 当前仍然依赖 history 返回中的最近一条失败记录

## 7. 风险与停手线

- 如果 latest failure entry hint 和现有 `latest_sample` 重复过多，会削弱 history 的紧凑性
- 如果提示开始解释为什么这条更重要，会和 summary 的导航字段重叠
- 因此本版应控制在“更快对上条目”而不是“追加新的判断层”
