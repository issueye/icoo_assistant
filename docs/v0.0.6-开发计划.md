# v0.0.6 开发计划

## 1. 版本定位

`v0.0.6` 聚焦把“任务和后台执行能看到关联”推进到“任务生命周期能获得基础自动维护”。核心目标是让后台执行结果对项目任务状态产生最小但清晰的影响。

## 2. 版本目标

本版本计划交付以下能力：

- 基于后台执行结果的任务状态联动策略
- 启动关联后台任务时的任务状态推进
- 后台完成后的基础状态回写约定
- 相关测试和文档

## 3. 范围定义

### 3.1 纳入 `v0.0.6`

- 项目任务与后台执行的基础状态联动
- 状态回写规则
- 文档与测试

### 3.2 暂不纳入 `v0.0.6`

- 自动任务拆解
- 自动认领任务
- 多 Agent 团队协作
- 审批协议
- worktree 绑定

## 4. 设计重点

### 4.1 联动原则

- 后台启动时可将关联任务推进到 `in_progress`
- 后台成功完成时只做最小回写，不强制自动 `completed`，除非规则明确
- 后台失败时应至少让任务上下文可见失败信息

### 4.2 版本重点

- 先做“任务状态可感知执行变化”
- 再做“完整自动编排”

## 5. 任务拆解

### A. 状态规则

- 设计后台执行与任务状态的最小联动规则
- 梳理失败和成功路径

### B. 代码实现

- 接入后台启动后的任务状态变化
- 接入后台完成后的任务上下文增强或状态回写

### C. 测试与文档

- 增加联动测试
- 更新 README
- 更新版本文档

## 6. 验收清单

- [x] 关联后台任务启动后可体现任务状态推进
- [x] 后台完成结果可回写到任务上下文或状态
- [x] 测试通过
- [x] 文档更新

## 7. 完成度评估

### 7.1 当前结论

`v0.0.6` 可以判定为已完成，项目任务已经具备围绕后台执行的最小自动状态维护能力。

完成依据：

- 已新增后台生命周期联动器
- 后台启动时会将关联 `pending` 任务推进到 `in_progress`
- 后台结束时会回写最近一次执行上下文
- 后台失败时会将 `in_progress` 任务退回 `pending`
- 已补充联动测试，`go test ./...` 通过
- README 已同步更新能力说明

### 7.2 当前边界

- 成功完成的后台执行不会自动将任务置为 `completed`
- 当前只保留最近一次后台执行上下文，尚未保留完整历史
- 自动联动规则仍保持克制，复杂编排留给后续版本
## 8. 风险提示

- 如果自动状态流转过于激进，容易让 Agent 失去对任务状态的控制感
- 失败路径处理需要兼顾可追溯性和后续人工修正
- 规则必须保持简单，避免在单 Agent 版本过早做复杂工作流引擎
