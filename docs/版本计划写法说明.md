# 版本计划写法说明

从 `v0.1.15` 起，版本计划改用更紧凑的写法，避免出现“结构重复、目标主观、粒度过细但文档很长”的问题。

## 核心原则

- 一个版本计划只回答本版真正新增了什么，不重复写长期不变的背景
- 验收标准优先写成可检查的产物，而不是“更顺滑”“更容易”这类纯主观描述
- 只在边界发生变化时写“本版不做”，不再机械重复整组长期非目标
- 如果版本只是某个主题里的一个字段级迭代，要明确说明它在主题链路中的位置
- 风险要对应当前版本真实引入的复杂度，而不是泛化表述

## 推荐结构

### 1. 版本主题

- 用一句话说明本版在解决什么具体问题

### 2. 为什么现在做

- 说明它和上一版能力的衔接关系
- 说明为什么不直接跳到更大的功能

### 3. 计划变更

- 直接列出本版预计新增的字段、行为、入口或测试点
- 尽量使用代码或文档中的真实名词

### 4. 本版不做

- 只写与当前版本最容易混淆、且确实不做的 1 到 3 项

### 5. 验收标准

- 使用客观可检查的条目，例如：
- `task_audit summary` 新增某字段
- README 新增某条演示命令
- `go test ./...` 通过

### 6. 完成度评估

- 说明是否完成
- 对照计划变更给出完成依据
- 补充当前边界

### 7. 风险与停手线

- 说明如果继续扩展最容易出现的复杂度
- 说明这一版为什么停在当前范围

## 继续下一版前的检查

- 先确认上一版是否已经满足“开发完成、测试通过、文档已更新、可提交”这四个条件
- 如果用户只说“继续下一个版本”，优先从上一版刚补上的能力里找一个最自然但尚未回答的问题
- 下一版主题必须能用一句话说清它和上一版的承接关系，不能只是在标题上做近义改写
- 如果候选内容同时会引入多个方向，先切最小的一块，避免一个版本混入多个主题
- 如果找不到清晰的增量点，应先停在评估或整理文档，而不是硬生生生出一个新版本

## 不推荐继续沿用的写法

- 每版都重复完全相同的“暂不纳入”列表
- 每版都保留相同的 A/B/C 任务拆解但没有新增信息
- 验收标准只写主观体验，不写产物或行为
- 计划主题只改几个词，但没有说明和上一版能力的关系
