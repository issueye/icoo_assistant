# v0.1.32 开发计划

## 1. 版本主题

收口首次上手路径里的最后一个高频命令误解：统一 CLI 默认展示的命令形式，让 README、`assistant check` 和 `--help` 都优先指向源码仓库里的 `go run ./cmd/assistant ...` 用法。

## 2. 为什么现在做

`v0.1.31` 已经统一了首次上手路径的步骤编号和“下一步从哪里继续”，但仍保留了一个真实摩擦点：README 主要使用 `go run ./cmd/assistant ...`，而 `assistant check` 与 `--help` 仍默认给出 `assistant ...` 形式。对于还没有安装二进制、只是直接克隆仓库运行的新用户，这会让第 1 步之后的下一条命令直接失真。

因此，`v0.1.32` 继续留在 post-release hardening，但只解决这一件事，不扩展成新的安装器或发布流程主题。

## 3. 计划变更

- 把 `assistant check` 中的 `minimal_happy_path` 和 `next_step` 默认命令切到 `go run ./cmd/assistant ...`
- 在自检输出里补充“如果已安装二进制，可把前缀替换成 assistant”的提示
- 把 `--help` 中的 `Usage`、`Examples` 和首次上手路径统一切到源码运行命令
- 把 fake mode 下的降级提示一并对齐到同样的命令形式
- 补充对应测试与 README 说明

## 4. 本版不做

- 安装器、打包分发或 PATH 自动检测
- 多平台专用启动脚本
- Team / Protocol / Worktree 主线功能

## 5. 验收标准

- [x] `assistant check` 默认输出 `go run ./cmd/assistant ...` 形式的最小 happy path
- [x] `assistant check` 明确提示二进制已安装时可替换成 `assistant`
- [x] `assistant --help` 与 fake mode 提示使用相同命令口径
- [x] README 已同步说明当前 CLI 默认命令形式
- [x] `go test ./...` 通过

## 6. 完成度评估

`v0.1.32` 可以判定为已完成。本版解决的是 `v0.1.31` 之后剩下的最后一个高频上手误解：用户按照 README 在源码仓库里执行 `go run ./cmd/assistant check` 后，CLI 却继续建议他执行 `assistant ...`，导致首次上手路径在第 2 步出现命令形式断层。

本版完成情况如下：

- `assistant check` 现在默认输出源码仓库命令 `go run ./cmd/assistant ...`
- 自检输出增加了命令前缀说明，并明确写出“如果二进制已安装，可替换成 assistant”
- `assistant --help` 的 `Usage`、`Examples` 和首次上手路径已经与 README 对齐
- fake mode 下的降级提示也改为默认推荐 `go run ./cmd/assistant check`
- CLI 相关测试已补齐并通过

当前边界判断：

- Stage A 的两版稳定化主题已经基本完成
- 当前仓库的首次上手路径、命令形式与 fake / real 模式边界已经比较清楚
- 下一版如果没有新发现的高频阻塞，就不应继续停留在 CLI 文案收口，而应切入 `v0.1.33` 的 Team 主线

## 7. 风险与停手线

- 如果继续在命令提示文案上反复微调，会重新回到低价值的小修小补
- 如果现在顺势扩展到安装器或平台专用脚本，会把版本主题从“减少误解”拉成“新增分发能力”
- 因此本版停在“统一默认命令口径并给出二进制替换提示”，不继续扩成更大的发布问题
