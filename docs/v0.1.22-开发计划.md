# v0.1.22 开发计划

## 1. 版本主题

让 `task_audit history` 在聚焦失败查看时补上一条直接的 latest failure command 提示，减少用户继续在条目里手动寻找本次执行命令。

## 2. 为什么现在做

`v0.1.21` 已经能让用户先定位到最新失败记录，下一步最自然的问题就是“这条最新失败到底执行了什么命令”。这一版继续停留在 history 可读性层，不扩展到新的推断逻辑。

## 3. 计划变更

- 为 `task_audit history` 增加一个轻量 latest failure command 提示
- 让失败历史视图更容易快速对齐当前失败记录与执行命令
- README 与 `tool_catalog` 同步更新
- 增加对应测试

## 4. 本版不做

- 命令级 diff
- 自动命令修复建议
- 更复杂的失败分组视图

## 5. 验收标准

- [x] `task_audit history` 新增 latest failure command 提示
- [x] README 与 `tool_catalog` 已更新
- [x] 相关测试补充完成
- [x] `go test ./...` 通过

## 6. 完成度评估

### 6.1 当前结论

`v0.1.22` 可以判定为已完成，当前任务侧审计已经能在 history 视图里通过 `latest_failure_command` 直接提示最新失败样本执行了什么命令，让用户在定位到最新失败后不必再继续在条目里手动找命令字段。

完成依据：

- 已在 `task_audit history` 中新增 `latest_failure_command`
- 当前 `status=failed` 与 `reason=<reason>` 的失败聚焦历史查看都会直接提示最新失败命令
- README 与 `tool_catalog` 已同步补上 latest failure command 提示说明
- 相关测试已补充并通过 `go test ./...`

### 6.2 当前边界

- 当前 `latest_failure_command` 仍然只是轻量提示，不承担命令解释
- 当前只提示最新失败命令，不扩展到命令级 diff
- 当前仍然依赖 history 返回中的最近一条失败记录

## 7. 风险与停手线

- 如果 latest failure command 提示和条目正文重复太多，会削弱 history 的紧凑性
- 如果提示开始附带命令解释，会和后续诊断层能力混在一起
- 因此本版应控制在“更快看到命令”而不是“解释命令为什么失败”
