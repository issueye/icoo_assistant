# 后续开发计划指南

## 1. 文档目的

本文档基于《当前项目功能与完成度评估报告》的结论，给出后续版本计划的选题原则、分期建议与止偏规则，避免后续小版本迭代方向继续散乱或回到机械微调。

## 2. 当前规划基线

根据评估结果，当前项目已经：

- 完成了单 Agent 本地闭环
- 完成了任务持久化、后台执行、审计和初代可用版收口
- 尚未真正进入多 Agent、协议化协作与 worktree 隔离阶段

因此，后续版本规划的核心原则不是“继续补边角字段”，而是“进入下一阶段主线”。

## 3. 后续版本计划的总原则

## 3.1 原则一：版本主题必须服务当前主线

从 `v0.1.31` 开始，每个版本都必须直接服务以下两个方向之一：

- `v0.1.30` 初代可用版的稳定化与真实上手摩擦修正
- 总规划中尚未展开的新阶段能力建设

如果一个版本既不提升稳定性，也不推进新阶段，就不应优先进入版本计划。

## 3.2 原则二：禁止默认沿着“小字段链路”机械迭代

以下类型的选题不应再作为默认主线：

- 再补一个 `task_audit history` 小提示字段
- 再补一个 README 里的同义表达
- 再补一个没有明确用户价值的轻量提示名词

只有当这类改动能直接解决真实上手阻塞、稳定性问题或新阶段关键阻塞时，才值得进入版本计划。

## 3.3 原则三：一个版本只回答一个明确问题

版本计划必须能用一句话说清楚“这一版解决的具体问题是什么”。

推荐问题形式：

- 用户第一次运行时最容易卡在哪里？
- 当前版本稳定使用时还缺哪一个关键入口？
- 进入多 Agent 协作前，最小基础设施还缺什么？

不推荐问题形式：

- 再做一点体验优化
- 再补几个说明
- 顺手把这些都做了

## 3.4 原则四：计划必须能被验证

每个版本都必须提供可检查的验收标准，例如：

- CLI 新增某个入口
- README 新增某段固定路径
- 某个模块目录与测试已落地
- `go test ./...` 通过

避免使用不可验证的表述，例如：

- 更顺滑
- 更智能
- 更容易理解

## 4. 后续版本选题优先级

## 4.1 第一优先级：v0.1.30 之后的轻量稳定化

优先处理这些问题：

- `assistant check`
- `--help`
- README
- fake/real 模式边界
- 最小 happy path 中仍容易误解的地方

这类版本要轻、要快、要有明确用户摩擦来源。

## 4.2 第二优先级：正式进入 Phase 5

当 `v0.1.30` 后的主要上手摩擦已经收口，就不应继续沉迷于文档与提示层优化，而应正式进入：

- Team Config
- teammate registry
- inbox / message bus
- lead -> teammate 最小通信链路

这会是后续版本真正的新主线。

## 4.3 第三优先级：Phase 6 协议与自治

只有在 Team 基础可运行之后，才应进入：

- request/response 协议
- 审批
- auto claim
- identity reinjection

否则容易做成没有承载对象的协议空壳。

## 4.4 第四优先级：Phase 7 Worktree Isolation

worktree 隔离必须后置，因为它依赖前面的任务系统、队友协作与执行边界已经足够清楚。

## 5. 推荐的后续阶段节奏

## 5.1 Stage A：Post-release hardening

适用版本：

- `v0.1.31`
- `v0.1.32`

目标：

- 修正初代可用版里的真实上手摩擦
- 不开启新大主题
- 为下一阶段腾出清晰边界

退出条件：

- `assistant check`
- `--help`
- README
- minimal happy path
- fake/real 模式边界

这些入口已经没有明显阻塞性问题

## 5.2 Stage B：Multi-Agent 基础设施

适用版本：

- `v0.1.33` 起

目标：

- 建立 `.team/` 基础结构
- 建立 teammate 生命周期最小骨架
- 建立消息收发最小闭环

退出条件：

- 至少两个 Agent 能稳定收发消息
- lead 与 teammate 关系可被查询和说明

## 5.3 Stage C：Protocol 与自治

目标：

- request/response 协议
- 审批
- auto claim
- idle polling

退出条件：

- 一个队友能在协议约束下认领、执行并响应

## 5.4 Stage D：Worktree 隔离执行

目标：

- worktree registry
- task-worktree binding
- isolated execution
- lifecycle event log

退出条件：

- 任务与隔离目录可以稳定绑定和回收

## 6. 每次制定新版本前的检查清单

在写下一个版本计划前，先逐项检查：

- 这个版本是否直接服务当前主线？
- 这个版本是否解决了一个真实的用户问题或阶段阻塞？
- 这个版本是否有清楚的验收标准？
- 这个版本是否会和前一版主题重复？
- 这个版本是否误回到了字段级机械迭代？

如果其中任意两项回答为“否”，说明这个版本主题需要重写。

## 7. 明确不推荐的规划方式

以下做法应避免继续使用：

- 沿着同一个小提示字段主题连续机械递增
- 一个版本同时混入文档修饰、功能补丁和下阶段预研
- 标题换词但问题没有变化
- 验收标准只写主观体验，不写可检查产物
- 明明已经该进入新阶段，却继续在旧主题里打转

## 8. 推荐的版本计划模板约束

后续版本计划建议强制包含这三个判断：

- 这一版解决的“唯一核心问题”
- 它和上一版之间的承接关系
- 它对当前主线的直接贡献

如果某个计划写不出这三点，就不应进入实际迭代。

## 9. 基于当前评估的直接建议

当前最合理的后续节奏是：

1. 先完成 `v0.1.31` 的 post-release hardening
2. 然后切入多 Agent 基础设施，不再继续默认围绕 `task_audit` 或 README 微调
3. 把 `v0.1.33` 之后的主线明确切到 Team / Protocol / Worktree

## 10. 一句话执行规则

后续每个小版本都必须满足一句话规则：

“它要么明显降低当前可用版的真实使用摩擦，要么明确推进最初总规划中尚未落地的下一阶段主线。”
